## 🏗️ 지갑 서비스 설계의 진화 과정

초당 100만 건 이상의 결제 명령($TPS$)을 처리하기 위해 시스템은 다음과 같은 단계를 거쳐 발전합니다.

### 1단계: 인메모리(Redis) 기반 설계

* **도입 이유:** $100$만 $TPS$라는 극단적인 성능 요구사항을 맞추기 위해 I/O가 가장 빠른 Redis와 샤딩 정보 관리를 위한 Zookeeper 조합을 선택.
* **한계 (정합성 결함):** **원자성(Atomicity) 보장 불가.** * A계좌(노드1)에서 차감 후, B계좌(노드2)에 입금하기 전 서비스 노드가 다운되면 **돈이 증발**하는 문제 발생.
* **한계 (내구성 부족):** 메모리 기반 저장소 특성상 노드 장애 시 데이터 유실 위험 상존.

### 2단계: 분산 트랜잭션 데이터베이스 기반 설계

* **도입 이유:** 1단계의 원자성 문제를 해결하기 위해 **ACID 트랜잭션**이 보장되는 RDB 도입 및 **2PC, TCC, Saga** 등의 프로토콜 검토.
* **한계 (감사 및 복구의 어려움):** 현재 잔액($Balance$)값만 업데이트하므로, "왜 이 잔액이 되었는가"에 대한 **데이터 감사(Audit)**가 불가능.
    * 정산 과정에서 불일치가 발견되어도 과거 이력을 역추적하여 복구하기가 매우 까다로움.

### 3단계: 이벤트 소싱(Event Sourcing) 도입

* **도입 이유:** 상태 변경 대신 모든 **변경 이력(Event)**을 순차적으로 저장하여 완벽한 추적성과 재현성 확보.
    * **성능 최적화:** 외부 DB 대신 **로컬 파일 시스템**에 Append-only 방식으로 기록하여 I/O 성능 극대화.
* **한계 (신뢰성 및 가용성):** 로컬 파일에만 의존할 경우 서버 장애 시 **단일 장애 지점(SPOF)**이 됨.
* **해결책:** **Raft 합의 알고리즘**을 도입하여 여러 노드에 이벤트를 복제함으로써 고가용성 달성.

### 4단계: CQRS 및 완성형 아키텍처

* **도입 이유:** 이벤트 소싱의 조회 병목을 해결하고 사용자 경험을 최적화하기 위함.
* **핵심 설계:**
    * **CQRS:** 쓰기(이벤트 저장)와 읽기(잔액 조회용 스냅샷) 모델을 분리하여 각 성능을 독립적으로 최적화.
    * **역방향 프록시:** 내부의 비동기 이벤트를 대기(Polling/Wait)하여 클라이언트에게는 동기식 응답으로 반환.

## 📚 주요 기술 키워드 이론 정리

### 1. 트랜잭션 프로토콜

* **2PC (Two-Phase Commit):** 모든 노드가 성공해야만 전체를 커밋하는 방식. 강력한 일관성을 보장하지만,  
  한 노드라도 응답이 없으면 전체 시스템이 대기 상태에 빠져 가용성이 떨어집니다.

* **TCC (Try-Confirm-Cancel):** 비즈니스 로직 차원의 2단계 커밋입니다. 먼저 자원을 예약(Try)하고,  
  성공 시 확정(Confirm), 실패 시 취소(Cancel) 로직을 명시적으로 구현합니다.

* **Saga (사가):** 각 서비스의 로컬 트랜잭션을 순차적으로 실행하며,   
  중간에 실패하면 이전에 완료된 작업들을 되돌리는 **보상 트랜잭션**을 실행하여 일관성을 맞추는 방식입니다.

### 2. 이벤트 소싱 (Event Sourcing)

데이터의 최종 상태만 저장하는 것이 아니라, 데이터에 일어난 **모든 변경 사항을 순서대로 저장**하는 방식입니다.

* **장점:** 완벽한 감사 이력(Audit Log) 제공, 특정 시점으로의 데이터 복구 용이.
* **특징:** 상태를 알려면 과거 이벤트를 재생(Replay)해야 하므로, 성능을 위해 '스냅샷' 기법을 병행합니다.

### 3. Raft 합의 알고리즘 (Raft Consensus)

* **정의:** 분산 시스템에서 노드 간 데이터 일치성을 보장하는 알고리즘.
* **작동:** 리더가 로그를 기록하고 과반수($Quorum$)의 복제 동의를 얻어야 커밋됨. 1단계에서 지적된 "노드 다운 시 데이터 불일치" 문제를 원천적으로 방지함.

### 4. CQRS (명령 및 조회 책임 분리)

* **정의:** 데이터 업데이트(Command)와 데이터 조회(Query)의 엔드포인트와 모델을 분리함.
* **이점:** 읽기 전용 DB를 별도로 두어 결제 처리 중에도 실시간 잔액 조회가 성능 저하 없이 가능함.

### 5. 역방향 프록시 및 동기/비동기 변환

* **정의:** 클라이언트 요청을 중계하며 내부의 비동기 메시지 기반 처리를 감추는 역할.
* **핵심:** 시스템 내부는 성능을 위해 비동기로 처리하되, 사용자가 받는 응답은 동기적으로 구성하여 결제 완료 여부를 즉시 알 수 있게 함.
