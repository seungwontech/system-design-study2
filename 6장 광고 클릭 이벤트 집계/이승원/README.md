# 광고 클릭 이벤트 집계

## 1. 요구사항 및 규모 추정

**트래픽**

- 일일 10억 건의 클릭 이벤트
- 평균 10,000 QPS / 최대 50,000 QPS

**주요 기능**

- 특정 광고(ad_id)의 분 단위 클릭 수 집계
- 최근 M분간 가장 많이 클릭된 상위 100개 광고 반환
- 국가, IP 등 다양한 속성별 필터링 지원

**데이터 규모**

- 월간 약 3TB의 원시 데이터 발생

---

## 2. 데이터 저장 전략: 원시 데이터 vs 집계 데이터

두 데이터의 **접근 패턴이 다르므로** 하이브리드 저장 방식을 채택합니다.

**1) 원시 데이터 (Raw Data)**

- 용도: 디버깅, 분석, 재계산
- 특징: 쓰기 중심, 막대한 용량
- DB: Cassandra 또는 S3 + 컬럼형 포맷

**2) 집계 데이터 (Aggregated Data)**

- 용도: 실시간 대시보드 조회
- 특징: 읽기/쓰기 모두 빈번, 시계열 데이터
- DB: Cassandra

**중요**: 버그 발생 시 집계 데이터가 오염될 수 있으므로, 원시 데이터를 활용해 다시 집계할 수 있도록 **반드시 보관**합니다.

---

## 3. 개략적 설계: 메시지 큐를 활용한 비동기 파이프라인

데이터 유실을 방지하고 생산자와 소비자의 결합을 끊기 위해 **카프카를 두 단계**로 활용합니다.

```
로그 수집기
    ↓
[첫 번째 큐] 원시 이벤트
    ↓              ↓
집계 서비스    원시 DB 저장
    ↓
[두 번째 큐] 집계 결과
    ↓
집계 DB 저장
    ↓
대시보드 조회
```

**각 단계 역할**

1. **첫 번째 큐**: 수집된 원시 이벤트를 담습니다
2. **집계 서비스**: 맵리듀스 프레임워크를 통해 분산 처리를 수행합니다
3. **두 번째 큐**: 집계된 결과(분 단위 합계 등)를 담아 DB에 원자적으로 커밋할 수 있게 합니다

---

## 4. 상세 설계: 집계 서비스와 윈도우 전략

### 4.1 맵리듀스 노드 구조

복잡한 집계 로직을 **DAG 모델**로 세분화합니다.

**맵(Map) 노드**

- 데이터를 읽어 정규화하고, ad_id를 기준으로 파티셔닝합니다

**집계(Aggregate) 노드**

- 메모리에서 ad_id별 클릭 수를 1분 단위로 합산합니다

**리듀스(Reduce) 노드**

- 각 집계 노드의 결과를 취합하여 최종 결과를 산출합니다

### 4.2 시간 관리와 윈도우

네트워크 지연으로 인해 **이벤트 발생 시각**과 **서버 도착 시각** 사이에 차이가 발생합니다.

**텀블링 윈도우 (Tumbling Window)**

- 매 분 고정된 주기로 클릭 수를 집계할 때 사용
- 예: `[00:00-01:00]` `[01:00-02:00]` (겹침 없음)

**슬라이딩 윈도우 (Sliding Window)**

- "지난 5분간 인기 광고"처럼 겹치는 구간의 통계에 적합
- 예: `[00:00-05:00]` `[01:00-06:00]` (겹침 있음)

**워터마크 (Watermark)**

- 지연된 이벤트를 처리하기 위해 윈도우를 일정 시간 더 열어두는 기술
- 트레이드오프: 정확도 ↑, 지연 시간 ↑

### 4.3 데이터 정확성 보장

중복 결제를 막기 위해 **정확히 한 번(Exactly-once) 전달** 방식을 구현합니다.

**오프셋 관리**

- 다운스트림에서 수신 확인(ACK)을 받은 후 오프셋을 외부 저장소에 기록
- 또는 분산 트랜잭션 내에서 처리하여 원자적 커밋을 보장

**프로세스**

```
1. 집계 수행
2. 다운스트림 전송
3. ACK 수신
4. 오프셋 저장  } 원자적 실행
5. 업스트림 커밋 }
```

### 4.4 확장성과 핫스팟 대응

**카프카 확장**

- ad_id를 해시 키로 사용하여 파티셔닝
- 지역별 토픽 샤딩 적용 (예: `topic_us`, `topic_eu`)

**핫스팟 문제**

- 상황: 특정 인기 광고에 클릭이 몰릴 경우
- 해결: 자원 관리자가 해당 노드에 추가 자원을 할당하고 데이터를 더 잘게 분할하여 처리

**예시**

```
집계 노드에 300개 이벤트 도착 (용량 100 초과)
    ↓
자원 관리자에 추가 노드 2개 요청
    ↓
이벤트를 3그룹으로 분할 (각 100개)
    ↓
병렬 집계 후 리듀스에서 재결합
```

## 5. 결함 내구성 및 모니터링

**체크포인트 (Checkpoint)**

- 집계는 메모리에서 이루어지므로 장애 시 복구를 위해 주기적으로 시스템 상태를 스냅숏으로 저장합니다

**조정 (Reconciliation)**

- 실시간 집계 결과의 정확성을 검증하기 위해 매일 밤 배치 작업을 돌려 실시간 결과와 대조 및 보정합니다

**스냅숏 포함 내용**

- 업스트림 오프셋 (어디까지 처리했는지)
- 분 단위 클릭 수 집계 상태
- Top N 광고 슬라이딩 윈도우 상태

|-------|---------------|-----------------|
| 아키텍처  | 카파 (단일 스트리밍)  | 코드 일원화, 재처리 간소화 |
