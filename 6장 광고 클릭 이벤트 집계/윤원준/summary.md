# 6장: 광고 클릭 이벤트 집계 시스템 설계

**설계의 트레이드오프, 면접 질문, 심화 토론**을 위주로 진행하기 위한 정리 자료입니다.

---

## 1. 핵심 요약
* **목표**: 매일 10억 건 이상의 광고 클릭 이벤트를 실시간으로 집계하여 광고주에게 과금 및 리포팅 제공.
* **핵심 컴포넌트**:
    * **Log Monitor/Collector**: 로그 파일에서 클릭 이벤트 수집.
    * **Message Queue (Kafka)**: 데이터 버퍼링 및 비동기 처리, 시스템 결합도 낮춤.
    * **Aggregation Service**: 실시간 스트림 처리 (MapReduce/DAG 모델 활용).
    * **Database**:
        * **Raw Data**: Cassandra/InfluxDB (쓰기 중심, 디버깅/재계산용).
        * **Aggregated Data**: Cassandra (읽기/쓰기 빈번, 시계열 데이터).
    * **Reconciliation**: 실시간 처리 결과와 배치 처리 결과를 비교하여 데이터 무결성 검증.
* **주요 특징**:
    * **Lambda/Kappa Architecture**: 실시간성과 정확성을 모두 잡기 위한 아키텍처 고려.
    * **Exactly-once Processing**: 과금과 직결되므로 중복 없는 정확한 처리가 필수.
    * **Event Time vs Processing Time**: 늦게 도착한 이벤트 처리를 위해 워터마크(Watermark) 기법 사용.

---

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)
설계 선택지 사이의 장단점을 비교하고, 상황에 맞는 최적의 선택을 논의합니다.

### 2.1. 데이터 처리 아키텍처: Lambda vs Kappa
| 구분 | **Lambda Architecture** | **Kappa Architecture** |
| :--- | :--- | :--- |
| **구조** | Batch Layer + Speed Layer (두 개의 경로) | Speed Layer Only (단일 경로) |
| **장점** | 배치 레이어를 통해 데이터 정확성 보장 및 복구 용이 | 코드 베이스가 하나라 유지보수가 쉬움 |
| **단점** | 로직을 두 번 구현해야 함 (Batch용, Stream용) | 데이터 재처리(Reprocessing) 시 스트림 처리 부하 발생 |
| **적합성** | **전통적인 빅데이터 시스템** | **최신 스트림 처리 엔진(Flink 등) 활용 시** |

> **논의 포인트**: 
> * "우리 팀 인력이 부족하다면?" (Kappa 아키텍처가 관리 포인트가 적어 유리)
> * "과거 1년치 데이터를 다시 집계해야 한다면?" (Lambda의 배치 레이어가 더 효율적일 수 있음)

### 2.2. 타임스탬프 기준: Event Time vs Processing Time
| 기준 | **Event Time (발생 시각)** | **Processing Time (처리 시각)** |
| :--- | :--- | :--- |
| **정의** | 사용자가 클릭한 실제 시각 | 서버가 이벤트를 수신/처리한 시각 |
| **장점** | **비즈니스적으로 정확함** (네트워크 지연 무관) | 구현이 매우 간단함 |
| **단점** | 늦게 도착한 데이터(Late Event) 처리 로직(워터마크 등) 필요 | 네트워크 지연 시 지표가 왜곡됨 |
| **적합성** | **광고 과금, 정확한 리포팅** | 시스템 부하 모니터링, 단순 로그 수집 |

> **논의 포인트**: 
> * "광고주에게 청구서를 보낼 때 어떤 시간이 기준이어야 할까?" (당연히 Event Time)

### 2.3. 데이터 저장 방식: Raw Data vs Aggregated Data
| 방식 | **Raw Data Only** | **Aggregated Data Only** | **Hybrid (둘 다 저장)** |
| :--- | :--- | :--- | :--- |
| **장점** | 데이터 손실 없음, 언제든 재집계 가능 | 저장소 비용 절감, 조회 속도 빠름 | **안정성과 성능 모두 확보** |
| **단점** | 저장 비용 막대함, 실시간 조회 느림 | 원본 데이터 유실 시 복구 불가능 | 비용이 가장 많이 듦 |
| **선택** | | | **본 설계안의 선택** |

## 3. 면접 예상 질문 (Interview Q&A)
면접관이 꼬리를 물고 들어올 수 있는 질문들과 모범 답변 키워드입니다.

### Q1. "중복된 클릭 이벤트(Exactly-once)는 어떻게 처리하나요?"
* **핵심 키워드**: `Idempotency(멱등성)`, `Offset 관리`, `분산 트랜잭션`
* **답변 논리**:
    * 클라이언트 측 중복은 고유 ID(Request ID)를 통해 걸러냅니다.
    * 서버 장애로 인한 재처리 중복은 **오프셋(Offset) 저장**과 **연산 상태 저장**을 하나의 **트랜잭션**으로 묶어서 처리하거나, 연산 자체를 **멱등(Idempotent)**하게 설계하여 여러 번 실행해도 결과가 같도록 합니다.

### Q2. "특정 광고(Ad)에 클릭이 몰리는 '핫스팟(Hotspot)' 문제는 어떻게 해결하나요?"
* **핵심 키워드**: `Local Aggregation`, `Sharding`, `Rebalancing`
* **답변 논리**:
    * 특정 파티션에 부하가 몰리면 처리 속도가 느려집니다.
    * **Local Aggregation**: 맵(Map) 단계에서 메모리 내에서 1차적으로 집계한 후 리듀스(Reduce)로 보냅니다.
    * **추가 샤딩**: 핫스팟이 발생한 키(Ad ID)에 임의의 접미사(Salt)를 붙여 여러 파티션으로 분산시킨 후, 나중에 다시 합칩니다.

### Q3. "늦게 도착한 데이터(Late Event)는 어떻게 처리하나요?"
* **핵심 키워드**: `Watermark`, `Windowing`, `Side Output`
* **답변 논리**:
    * **워터마크(Watermark)**를 설정하여 일정 시간(예: 1분)까지는 기다려줍니다.
    * 워터마크 이후에 도착한 데이터는 버리거나, 별도의 **Side Output**으로 빼서 나중에 배치로 보정(Reconciliation)합니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)
단순한 구현을 넘어 시스템의 확장성과 엣지 케이스를 고민해봅니다.

### 주제 A: 광고 사기(Ad Fraud) 탐지와 실시간 필터링
* **문제**: 봇(Bot)이나 경쟁사가 악의적으로 클릭을 발생시켜 광고비를 소진시킨다면?
* **접근법**:
    * 단순 집계 전에 **Fraud Detection** 컴포넌트를 둡니다.
    * IP 블랙리스트, 클릭 패턴 분석(너무 빠른 클릭), 머신러닝 모델 등을 통해 실시간으로 필터링합니다.

### 주제 B: 데이터 재처리(Reprocessing) 전략
* **문제**: 집계 로직에 버그가 있어서 지난 3일치 데이터를 다시 계산해야 한다면?
* **접근법**:
    * **Raw Data**가 저장되어 있어야 합니다.
    * 실시간 파이프라인과는 별도의 **재처리 전용 파이프라인**을 띄워서 운영 데이터에 영향을 주지 않고 빠르게 처리한 후, 결과 DB를 덮어쓰거나 보정합니다.

### 주제 C: 스타 스키마(Star Schema)와 OLAP 활용
* **문제**: 광고주가 "지역별", "성별", "OS별" 등 다양한 조합으로 리포트를 보고 싶어 한다면?
* **접근법**:
    * 미리 모든 조합을 집계하는 것은 불가능합니다.
    * **ClickHouse, Druid, Pinot** 같은 OLAP 데이터베이스를 도입하여, 대량의 데이터를 실시간으로 다차원 분석할 수 있도록 합니다.

## 5. 추가 학습 추천
* **Apache Flink**: 상태 기반 스트림 처리(Stateful Stream Processing)의 사실상 표준. 워터마크, 윈도우, Exactly-once 등을 깊게 이해할 수 있음.
* **Apache Druid / ClickHouse**: 실시간 분석을 위한 OLAP 데이터베이스.
* **Bloom Filter**: 중복 클릭 방지나 집합 멤버십 확인을 위한 확률적 자료구조.
* **HyperLogLog**: 대규모 데이터의 카디널리티(유니크 방문자 수 등)를 적은 메모리로 추정하는 알고리즘.
