# 11장: 결제 시스템 핵심 요약 및 심화 분석

## 1. 핵심 요약

결제 시스템은 처리 대역폭(TPS)보다 **'정확히 한 번 처리'와 '시스템 간 데이터 정합성'**이 최우선 되는 특수한 도메인입니다.

### 1) 전체 시스템 흐름
전자상거래 결제 시스템은 돈의 흐름을 기준으로 크게 두 단계로 나뉩니다.
*   **대금 수신 (Pay-in):** 구매자로부터 결제 대금을 수령하여 전자상거래 업체의 은행 계좌로 입금합니다.
*   **대금 정산 (Pay-out):** 수령한 대금에서 플랫폼 수수료를 제외한 최종 수익 금액을 판매자의 은행 계좌로 송금합니다.
*   *결제 시스템은 위 흐름을 완수하기 위해 계정, 지갑, 원장 등의 내부 서비스는 물론 결제 서비스 공급자(PSP)와 지속적으로 통신하고 연동합니다.*

### 2) 주요 컴포넌트 및 데이터베이스 설계
강력한 법적, 보안적 규제 때문에 우리 결제 시스템은 민감한 신용 카드 번호를 직접 수집하거나 저장하지 않습니다.
*   **결제 서비스 및 실행자:** 사용자의 결제 이벤트를 수신하면 외부 **PSP(Stripe 등)**를 호출하여 카드 승인을 받아옵니다.
*   **지갑 (Wallet):** 특정 판매자 혹은 사용자의 잔고(Balance)를 기록하는 장부입니다.
*   **원장 (Ledger)과 복식부기:** 결제 시스템의 가장 핵심적인 금융 기록입니다. **복식부기(Double-entry) 원칙**에 따라 모든 거래는 돈이 빠져나가는 계좌와 들어오는 계좌 양쪽에 동일한 금액으로 기록되어 총합이 무조건 '0'이 되어야 합니다.
*   **저장소 선택 (RDBMS):** 위와 같은 복잡한 금융 트랜잭션의 원자성과 일관성(ACID)을 강력히 보장하기 위해 NoSQL보다 **수십 년간 금융권에서 검증된 RDBMS**를 우선적으로 선택합니다.

### 3) 외부 결제 페이지 연동 메커니즘
우리 시스템이 카드 번호를 묻지 않으면서 결제를 안전하게 끝내는 구체적인 절차입니다.
결제 서비스는 결제 금액 등 필수 정보를 담아 PSP에 전송해 고유한 `토큰(Token)`을 받아옵니다. 클라이언트는 이 토큰을 사용해 PSP가 직접 제공하는 외부 결제창(보통 iframe이나 SDK 형태)을 화면에 띄웁니다. 사용자가 PSP 창에 카드를 긁고 결제가 성공하면, PSP는 비동기 **웹훅(Webhook)**을 우리 서버로 쏘아 보내 결제가 완료되었음을 알려줍니다.

### 4) 가장 중요한 요구사항: '정확히 한 번 전달' 보장
결제 시스템에서 가장 치명적인 장애인 '고객에게 중복으로 요금이 청구되는 일(이중 결제)'을 막기 위해 두 가지 방어벽을 설계합니다.

*   **최소 한 번 실행 (Retry & DLQ):** 외부 통신 과정에서 일시적인 네트워크 장애가 나면 포기하지 않고 지수적 백오프(Exponential Backoff) 방식을 써서 결제를 재시도합니다. 만약 정해진 횟수를 넘긴 영구적 실패라면, 데드 레터 큐(실패 MQ)로 이벤트를 격리하여 알림을 울리고 수동 조사에 들어갑니다.
*   **최대 한 번 실행 (멱등성, Idempotency):** 사용자가 결제 버튼을 실수로 다다닥 중복 클릭하더라도 하나만 처리해야 합니다. 클라이언트는 결제 요청을 보낼 때 중복되지 않는 난수인 `멱등 키`를 함께 보냅니다. 우리 DB는 이 키를 고유 제약 조건(Unique Key)으로 관리하므로, 동일한 멱등 키가 또 들어오면 에러를 뱉어내거나 이미 처리된 이전 결과만 돌려주어 이중 결제를 완벽히 차단합니다.

### 5) 최후의 방어선: 데이터 상태 조정 (Reconciliation)
비동기 통신이 주를 이루기 때문에 네트워크에서 웹훅 유실, 메시지 큐 지연 등으로 인해 우리 원장과 외부 PSP 간의 잔액 기록이 어긋날 위험이 언제나 도사리고 있습니다. 
이를 바로잡기 위해 매일 밤 PSP나 은행이 오늘 하루 일어난 거래 내역을 정리한 **정산 파일(Settlement File)**을 다운받습니다. 이 파일을 우리 시스템의 원장 데이터와 대조하여 누락되거나 틀린 건이 없는지 검사하고 불일치분을 교정하는 배치(Batch) 작업이 반드시 필요합니다. 이를 상태 조정(Reconciliation) 시스템이라고 부릅니다.

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)

*   **결제 정보 직접 처리 vs 외부 결제 페이지 (PSP) 위임:**
    *   *직접 처리:* 완전한 맞춤형 UX 제공 및 데이터 통제가 가능하지만, 엄청난 비용을 들여 엄격한 PCI-DSS 규정을 통과해야 하며 해킹 등 보안 리스크를 우리가 전부 짊어집니다.
    *   *PSP 위임:* 보안 및 법적 부담을 PSP에 넘기고 우리는 핵심 비즈니스 로직에만 집중할 수 있습니다. 반면, 사용자 경험이 외부 페이지에 다소 종속되며 웹훅 처리나 폴링(Polling) 같은 비동기 동기화 로직 때문에 시스템 복잡도가 크게 증가합니다.
*   **동기식 통신 vs 비동기 통신 (메시지 큐):**
    *   *동기식 통신(HTTP):* 설계와 디버깅이 직관적입니다. 하지만 PSP 연동 지연 발생 시 우리 서버의 스레드 자원이 고갈되며, 시스템 간 강하게 결합되어 있어 하나의 서비스가 죽으면 전체 결제가 실패로 이어지는 연쇄 장애에 취약합니다.
    *   *비동기 통신(Kafka 등):* 서비스 간 결합도를 끊어내어 처리율과 안정성을 높입니다. 결제 완료 이벤트를 큐에 던지면, 지갑, 알림, 분석 등 각 서비스가 자신의 페이스에 맞게 이벤트를 소비합니다. 단, 전체 시스템의 강력한 일관성을 실시간으로 맞추기 어려워 최종 일관성(Eventual Consistency) 모델과 조정(Reconciliation) 시스템에 의존해야 합니다.
*   **데이터 일관성 모델 (복제 지연 문제):**
    *   장애 대비를 위해 데이터베이스를 다중화(Master-Replica)했을 때, 복제 지연 시간 찰나에 조회 쿼리가 들어오면 결제 상태 불일치가 발생할 수 있습니다.
    *   가장 쉬운 해결책은 주(Master) DB에서만 읽기/쓰기를 모두 처리하는 것이지만 자원이 비효율적으로 쓰입니다. 반대로 성능을 위해 사본을 적극 활용하려면 Spanner와 같이 합의 알고리즘에 기반한 비싼 분산 DB를 도입해야 하는 설계적 딜레마가 존재합니다.

## 3. 면접 예상 질문 (Interview Q&A)

**Q. 사용자가 '결제' 버튼을 클릭한 직후 네트워크 오류가 생겨서 다시 '결제' 버튼을 눌렀습니다. 결제 시스템은 이중 결제를 어떻게 방지할 수 있습니까?**
A. 결제 시스템은 어떤 경우에도 '멱등성(Idempotency)'을 보장해야 합니다. 클라이언트가 결제 요청을 생성할 때 고유한 '멱등 키'를 생성하여 요청 헤더에 함께 담아 보냅니다. 결제 DB에서는 이 키를 Unique 제약 조건으로 설정하고 있습니다. 따라서 동일한 키로 DB 삽입이 또 시도되면 시스템은 중복 요청으로 간주하고 처리를 거부하거나 최초의 정상 처리 결과를 그대로 반환하여 이중 청구를 막아냅니다.

**Q. 외부 PSP(결제 서비스 공급자)와 연동할 때 통신 장애 등으로 데이터 불일치 문제가 발생할 수 있는데, 어떻게 해결합니까?**
A. 비동기 웹훅 기반 환경에서는 웹훅 메시지를 놓치거나 늦게 받는 일이 잦습니다. 이를 대비해 두 가지 전략을 혼용합니다. 첫째, 웹훅만 기다리지 않고 우리 시스템이 주기적으로 PSP에 현재 결제 상태를 묻는 폴링(Polling) 스케줄러를 가동합니다. 둘째, 최후의 방어선으로 매일 밤 PSP가 제공하는 전체 정산 파일과 우리 시스템의 원장 데이터를 대조하는 조정(Reconciliation) 배치 작업을 돌려 최종적인 데이터 정합성을 맞춥니다.

**Q. 결제 도메인에서 저장소로 NoSQL보다 RDBMS를 권장하는 가장 핵심적인 이유는 무엇입니까?**
A. 결제 시스템은 처리량(성능)보다 ACID(원자성, 일관성, 고립성, 지속성) 트랜잭션 보장이 절대적으로 중요합니다. 복식부기 원칙에 따라, 한 사용자의 지갑에서 돈이 차감되고 원장에 그 내역이 정확히 기록되는 두 과정이 완벽히 한 묶음의 트랜잭션으로 수행되거나 실패해야 하기 때문입니다. 또한, RDBMS는 금융업계에서 수십 년간 안정성이 검증되었고 전문 DBA 풀이 풍부하다는 운영 상의 결정적 장점이 있습니다.

**Q. 외부 결제사의 장애 등 결제 실패 시, 우리 시스템 내에서 이를 어떻게 안전하게 처리합니까?**
A. 일시적인 네트워크 오류인지 영구적 실패인지 구분하여 처리합니다. 일시적 오류라면 '재시도 큐(Retry Queue)'를 두고, 외부 시스템에 과부하를 주지 않도록 지수적 백오프(Exponential Backoff) 전략을 써서 점진적으로 간격을 늘려가며 재시도합니다. 정해진 재시도 횟수를 넘겨버린 심각한 실패 건은 '데드 레터 큐(Dead Letter Queue)'로 보내어 로직 수행을 중단시키고, 개발자나 재무팀이 수동으로 원인을 분석할 수 있도록 격리합니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)

*   **MSA 환경에서의 분산 트랜잭션 (Saga Pattern):** 주문, 결제, 재고 차감, 포인트 적립이 각각 별개의 마이크로서비스 DB로 쪼개져 있을 때, 이들을 어떻게 하나의 원자적(Atomic) 트랜잭션처럼 묶을 것인가? 코레오그래피(Choreography) 이벤트 기반 방식과 오케스트레이터(Orchestrator) 지휘 방식의 차이점 및 실패 시 되돌리는 보상 트랜잭션(Compensating Transaction) 설계 기법.
*   **카프카(Kafka) 기반 Exactly-Once 시맨틱:** 다중 수신자 환경에서 결제 완료 이벤트를 뿌릴 때, 카프카 프로듀서와 컨슈머 양단에서 메시지가 중복 발행되거나 2번 소비되지 않도록 보장하는 트랜잭션 기능과 멱등성 컨슈머(Idempotent Consumer) 구현 원리.
*   **복식부기(Double-entry) 원장 DB 스키마 병목 완화:** 실제 금융권이나 거대 핀테크 플랫폼에서 복식부기를 RDBMS로 구현할 때, 인기 있는 판매자 계좌로 엄청난 트랜잭션이 몰리며 발생하는 Row 잠금(Row Lock) 경합 문제. 이를 완화하면서도 계좌 총계 잔고 무결성을 빠르게 계산하기 위한 샤딩이나 인메모리 배칭 기법.

## 5. 추가 학습 추천
*   **Stripe / Toss API 문서:** 업계 표준이 되는 세계구급 결제 플랫폼의 API 가이드에서 결제 흐름과 특히 `Idempotency-Key` 파라미터를 실무에서 어떻게 설계하고 활용하는지 확인해 보기.
*   **Kafka Exactly-Once Semantics (KIP-98):** 대용량 분산 메시징 시스템에서 '정확히 한 번 전달'이 수학적, 시스템적으로 어떻게 멱등성과 트랜잭션 로그를 통해 보장되는지 그 깊은 이론적 원리 파악.
*   **분산 데이터베이스(Spanner, CockroachDB):** 무제한에 가까운 스케일아웃을 지원하면서도 강력한 RDBMS 정합성을 놓치지 않기 위해 클록 동기화(TrueTime) 기술이나 합의 알고리즘(Raft, Paxos)을 어떻게 활용하는지 연구해 보기.
