# (11) 결제 시스템

## 1단계: 문제 이해 및 설계 범위 확정

신용 카드 결제, 하나의 통화로 한정

### 기능 요구사항

- 대금 수신(pay-in) 흐름: 결제 시스템이 판매자 대신 고객으로 부터 대금 수령
- 대금 정산(pay-out) 흐름: 결제 시스템이 전 세계의 판매자에게 제품 판매 대금을 송금

### 비기능 요구사항

- 신뢰성 및 내결함성: 결제 실패는 신중하게 처리해야 함
- 내부 서비스(결제 시스템, 회계시스템) ↔ 외부 서비스(결제 서비스 제공업체) 간의 조정 프로세스: 시스템 간의 결제 정보가 일치하는지 비동기적으로 확인 필요

### 개략적인 규모 추정

- 하루 100만 건의 거래(트랜잭션) 가정 ⇒ 초당 10건의 TPS
- 일반적인 데이터베이스로 처리 가능 → 처리 대역폭 대신 트랜잭션의 정확성에 초점

## 2단계: 개략적 설계안 제시 및 동의 구하기

```sql
		구매자        전자상거래웹사이트             판매자
		  ㅣ								ㅣ								    ㅣ
	신용카드 -대금수신->	    은행계좌    -대금정산-> 은행계좌
```

### 대금 수신 흐름

![image.png](https://possible-telephone-196.notion.site/image/attachment%3A80fefb70-ced7-4db9-8c28-677431d0bc58%3Aimage.png?table=block&id=335be4ca-d9d2-80ad-bd60-f4599c2fced6&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=940&userId=&cache=v2)

- 결제 서비스 (payment service)
    - 사용자로부터 결제 이벤트를 수락, 결제프로세스를 조율
    - 일반적으로 가장 먼저 하는 일: 위험 점검(risk check)
        - AML/CFT와 같은 규정 준수 체크
        - 자금 세탁, 테러 자금 조달 등의 범죄 행위 증거 평가
    - 고도로 전문화 되어 있어 보통 제 3자 제공업체 사용
- 결제 실행자 (payment executor)
    - 결제 서비스 공급자(PSP)를 통해 결제 주문 하나를 실행 (하나의 결제 이벤트에는 여러 결제 주문이 포함될 수 있음)
- 결제 서비스 공급자 (PSP, Payment Service Provider)
    - A 계정에서 B 계정으로 돈을 옮기는 역할 (구매자의 신용 카드 계좌에서 돈을 인출하는 역할)
- 카드 유형 (카드사)
    - 비자, 마스터카드, 디스커버리 등
- 원장 (ledger)
    - 결제 트랜잭션에 대한 금융 기록
    - ex) 사용자가 판매자에게 1달러 결제 → 사용자로부터 1달러 인출 & 판매자에게 1달러 지급
    ⇒ 위 내용을 기록으로 남김
    - 결제 후 분석에 매우 중요한 역할
- 지갑 (wallet)
    - 판매자(merchant)의 계정 잔액을 기록
    - 특정 사용자가 결제한 총 금액도 기록 가능
- 결제의 흐름
    1. 사용자: ‘주문하기’ 버튼 클릭 → 결제 이벤트가 생성되어 결제 서비스로 전송
    2. 결제 서비스: 결제 이벤트를 데이터베이스에 저장
    3. 결제서비스: 단일 결제 이벤트에 여러 결제 주문이 포함(다건 결제) 된 경우 결제 주문마다 결제 실행자를 호출
    4. 결제 실행자: 결제 주문을 데이터베이스에 저장
    5. 결제 실행자: 외부 PSP 호출, 신용 카드 결제 처리
    6. 결제 서비스: 결제 실행자가 결제를 성공적으로 처리 → 지갑을 갱신. 특정 판매자의 잔고 기록
    7. 지갑 서버: 갱신된 잔고 정보를 데이터베이스에 저장
    8. 결제 서비스: 지갑 서비스가 판매자 잔고를 성공적으로 갱신하면 원장 호출
    9. 원장 서비스: 새 원장 정보를 데이터베이스에 추가
- 결제 서비스 API
    - POST /v1/payments
        - 결제 이벤트를 실행
        - 입력 인자
            
            
            | 필드 | 설명 | 자료형 |
            | --- | --- | --- |
            | buyer_info | 구매자 정보 | json |
            | checkout_id | 해당 결제 이벤트를 식별하는 전역적으로 고유한 ID | string |
            | credit_card_info | 암호화된 신용 카드 정보 또는 결제 토큰, PSP마다 다른 값 | json |
            | payment_orders | 결제 주문 목록 | list |
        - payment_orders
            
            
            | 필드 | 설명 | 자료형 |
            | --- | --- | --- |
            | seller_account | 대금을 수령할 판매자 | string |
            | amount | 해당 주문으로 전송되어야 할 대금 | string |
            | currency | 주문에 사용된 통화 단위 | string(ISO 4217) |
            | payment_order_id | 해당 주문을 식별하는 전역적으로 고유한 ID | string |
    - GET /v1/payments/{:id}
        - payment_order_id가 가리키는 단일 결제 주문의 실행 상태 반환
- 결제 서비스 데이터 모델
    - 결제 시스템용 저장소 솔루션을 고를 때의 고려사항
        1. 안정성이 검증되었는가? : 다른 대형 금융 회사에서 수년(가령 5년 이상) 동안 긍정적인 패드백을 받았는가?
        2. 모니터링 및 데이터 탐사(investigation)에 필요한 도구가 풍부하게 지원되는가?
        3. 데이터 베이스 관리자(DBA) 채용 시장이 성숙했는가? = 숙련된 DBA를 쉽게 채용할 수 있는가?
        - 일반적으로 NoSQL/NewSQL보다는 ACID 트랜잭션을 지원하는 전통적인 RDBMS를 선호
    - 결제 이벤트 테이블
        
        
        | 필드명 | 타입 | 제약조건 |
        | --- | --- | --- |
        | checkout_id | string | PK |
        | buyer_info | string | - |
        | seller_info | string | - |
        | credit_card_info | 카드 제공업체에 따라 다름 | - |
        | is_payment_done | boolean | - |
    - 결제 주문 테이블
        
        
        | 필드명 | 타입 | 제약조건 |
        | --- | --- | --- |
        | payment_order_id | string | PK |
        | buyer_account | string | - |
        | amount | string | - |
        | currency | string | - |
        | checkout_id | string | FK |
        | payment_order_status | string | - |
        | ledger_updated | boolean | - |
        | wallet_updated | boolean | - |
        - payment_order_status: 결제 주문의 상탤ㄹ 유지하는 열거 자료형(enum)
            1. 초기값: NOT_STARTED
            2. 결제 서비스: 결제 실행자에게 주문 전송: EXECUTING으로 변경
            3. 결제 서비스: 결제 처리자의 응답에 따라 SUCCESS 또는 FAIL 로 변경
        - payment_order_status의 값이 SUCCESS로 결정 -> 결제 서비스: 지갑 서비스를 호출해 판매자 잔액 업데이트, wallet_updated필드의 값: TRUE로 갱신
        - → 결제서비스: 원장 서비스를 호출하여 원장 데이터베이스의 ledger_updated 필드를 TRUE로 갱신
        - → 동일한 checkout_id 아래의 모든 결제 주문이 성공적으로 처리 → 결제 서비스: is_paymnent_done을 TRUE로 갱신
    - +) 종결되지 않은 결제 주문을 모니터링하는 작업(scheduled job)을 만들어 임계값으로 설정된 기간이 지나도록 완료되지 않은 결제 주문이 있는지 모니터링
- 복식부기(double_entry) 원장 시스템
    
    
    | 계정 | 차감 | 증가 |
    | --- | --- | --- |
    | 구매자 | $ 1 |  |
    | 판매자 |  | $ 2 |
    - 모든 결제 시스템에 필수 요소. 정확한 기록을 남기는데 핵심적 역할
    - 모든 거래 항목의 합계는 0이어야 함.

### 외부 결제 페이지

- 대부분의 기업은 신용 카드 정보를 취급하지 않기 위해(복잡한 규정) PSP에서 제공하는 외부 신용 카드 페이지를 사용.
    - 웹사이트의 경우, 위젯이나 iframe. 모바일의 경우, 결제 SDK에 포함된 사전에 구현된 페이지 으로 외부 페이지 구현

## 3단계: 상세 설계

시스템 오류, 장애 대응 위주

- PSP 연동 (성공적인 흐름)
    
    ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A3dbfce36-7b34-494b-bf9a-efcde7ba2426%3Aimage.png?table=block&id=335be4ca-d9d2-80bb-af6d-d88b6b4adf25&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=860&userId=&cache=v2)
    
    - 결제 시스템을 PSP와 연동하는 두가지 방법
        - 회사가 민감한 결제 정보를 안전하게 저장할 수 있는 경우: API를 통해 PSP와 연동.
            - 회사: 결제 웹페이지 개발, 민감한 결제 정보 수집
            - 은행 연결, 다양한 카드 유형 지원
        - 복잡한 규정 및 보안문제로 결제 정보 저장하지 않는 경우: PSP는 카드 결제 세부 정보를 수집, PSP에 안정하게 저장할 수 있도록 외부 결제 페이지를 제공
    - 외부 결제 페이지 이용 흐름 (결제 실행자, 원장, 지갑 생략)
        
        ![image.png](https://possible-telephone-196.notion.site/image/attachment%3Af3912b45-11a2-461f-92ff-4a85041c6c4f%3Aimage.png?table=block&id=335be4ca-d9d2-80cc-9fbf-d76ba25e9f89&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=780&userId=&cache=v2)
        
        1. 사용자: 클라이언트 브라우저에서 ‘결제’ 버튼 클릭 → 클라이언트: 결제 주문 정보를 담아 결제 서비스 호출
        2. 결제 서비스: 결제 주문 정보 수신, 결제 등록 요청을 PSP로 전송
            - 등록 요청: 결제 금액, 통화, 결제 요청 만료일, 리디렉션 URL 등의 결제 정보 포함
            - 결제주문이 정확히 한 번 등록 될 수 있도록 UUID 필드 사용
        3. PSP: 결제 서비스에 토큰(UUID) 반환
        4. 결제 서비스: 토큰을 데이터베이스에 저장
        5. 클라이언트: PSP가 제공하는 외부 결제 페이지를 표시
            - 예시: 스트라이프(PSP) 사의 웹 연동 사례. 민감 결제 정보는 스트라이프 사가 수집, 호출 페이지로 넘어오지 않음
                
                ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A7d3ddea9-379b-441c-8413-b7a4c8b5bd17%3Aimage.png?table=block&id=335be4ca-d9d2-8021-9289-dc3fc2d5e48f&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=630&userId=&cache=v2)
                
                - 외부 결제 페이지가 필요해 하는 두가지 정보
                    1. 4단계에서 받은 토큰: PSP의 자바스크립트 코드는 이 토큰을 사용하여 PSP의 백엔드에서 결제 요청에 대한 상세 정보를 검색. (ex. 사용자에게서 받을 금액)
                    2. 리디렉션 URL: 결제가 완료되면 호출될 웹 페이지 URL. PSP의 자바스크립트는 결제가 완료되면 브라우저를 리디렉션 URL로 돌려보냄. 
        6. 사용자: 신용 카드 번호, 소유자 이름, 카드 유효기간 등의 결제 세부 정보를 PSP의 웹 페이지에 입력. 결제 버튼 클릭 → PSP가 결제 처리를 시작
        7. PSP: 결제 상태를 반환
        8. 사용자: 리디렉션 URL이 가리키는 웹페이지로 보내짐 (7단계에 수신 된 결제 상태가 URL에 추가되곤 함)
            
            ```sql
            --ex. 
            https://your-company.com/?tokenID=JIOUA210293189FJDS&payResult=X324FSa
            ```
            
        9. PSP: 비동기적으로 웹훅을 통해 결제 상태와 함께 결제 서비스를 호출
        → 결제 서비스: 웹훅을 통해 결제 이벤트를 다시 수신. 결제 상태를 추출하여 주문 데이터벵스 테이블의 payment_order_status 필드를 최신 상태로 업데이트
            - 웹훅: 결제 시스템 측에서 PSP를 처음 설정할 때 등록한 URL
- 조정(reconciliation)
    - 조정: 비동기 통신을 할 경우 관련 서비스 간의 상태를 주기적으로 비교하여 일치하는지 확인하며 정확성을 보장. 일반적으로 결제 시스템의 마지막 방어선으로 받아들여 짐.
        
        ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A6c87ea6d-8805-4fd4-91b1-e8b340a6667b%3Aimage.png?table=block&id=335be4ca-d9d2-80af-823b-f6f13919f215&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=780&userId=&cache=v2)
        
    - 매일 밤 PSP나 은행 → 고객에게 정산(settlement)파일을 보냄. 정산 파일: 은행 계좌의 잔액과 하루 동안 해당 계좌에서 발생한 모든 거래 내역기 기재
    - 조정 시스템: 정산 파일의 세부 정보를 읽어 원장 시스템과 비교
    - 원장과 지갑의 상태가 같은지 등 결제 시스템의 내부 일관성도 확인.
    - 발생 가능한 불일치 및 해결 방안 세가지
        1. 어떤 유형의 문제인지 알고 있으며, 문제 해결 절차를 자동화 할수 있는 경우: 불일치 문제의 분류 및 조정 작업 자동화 가능
        2. 어떤 유형의 문제인지는 알지만 문제 해결 절차를 자동화 할 수 없는 경우: 자동 조정 프로그램의 작성 비용이 너무 높거나 하는 경우 → 작업 대기열에 넣고 재무팀에서 수동으로 수정하도록
        3. 분류할 수 없는 유형의 문제인 경우: 특별 작업 대기열에 넣고 재무팀에서 조사하도록
- 결제 지연 처리
    - 결제 요청이 평소보다 오래 걸리게 되는 사례
        1. PSP가 해당 결제 요청의 위험성이 높다고 보고 담당자 검토를 요구하는 경우
        2. 신용 카드사가 구매 확인 용도로 카드 소유자의 추가 정보를 요청하는 3D보안 인증 같은 추가 보호 장치를 요구하는 경우
    - 구매 페이지가 외부 PSP에 호스팅 되는 경우 PSP가 처리하는 방법
        - PSP: 결제가 대기(pending) 상태임을 알리는 상태 정보를 클라이언트에 반환, 클라이언트는 이를 사용자에 표시 및 현재 결제 상태를 확인할 수 있는 페이지 제공
        - PSP는 판매사를 대신하여 대기 중인 결제의 진행 상황을 추적, 상태가 바뀌면 PSP에 등록된 웹 훅을 통해 결제 서비스에 알림
            - 웹훅 대신 폴링을 사용하기도 함
- 내부 서비스 간 통신
    - 동기식 통신
        - 소규모 시스템에서는 잘 작동하지만 규모가 커지면 생기는 단점들
            - 성능 저하: 요청 처리에 관계된 서비스 가운데 하나에 발생한 성능 문제가 전체 시스템의 성능에 영향
            - 장애 격리 곤란: PSP 등의 서비스에 장애가 발생하면 클라이언트는 더이상 응답을 받지 못함
            - 높은 결합도: 요청 발신자는 수신자를 알아야만 한다.
            - 낮은 확장성: 큐를 버퍼로 사용하지 않고서는 갑작스러운 트래픽 증가에 대응할 수 있도록 시스템을 확장하기 어려움
    - 비동기식 통신
        - 단일 수신자
            
            ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A0eb3aee6-13f9-4b69-a9ee-b40dbdbb422b%3Aimage.png?table=block&id=335be4ca-d9d2-8031-89a4-f3b7699e382e&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=710&userId=&cache=v2)
            
            - 각 요청(메시지)를 하나의 수신자 or 서비스가 처리.
            - 공유 메시지 큐를 사용해 구현
            - 큐에는 수의 구독자가 있을 수 있으나 처리된 메시지는 큐에서 바로 제거 됨
        - 다중 수신자
            
            ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A3514ae4f-79a3-4b88-bacc-b8a7ba31bf32%3Aimage.png?table=block&id=335be4ca-d9d2-8015-8bfe-d96437e9617b&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=710&userId=&cache=v2)
            
            - 각 요청(메시지)를 여러 수신자 or 서버가 처리함.
            - 수신자가 수신한 메시지는 큐(카프카)에서 바로사라지지 않아, 동일한 메시지를 여러 서비스가 받아서 처리 가능
            - 하나의 요청 → 푸시 알림 전송, 재무 보고 업데이트, 분석 결과 업데이트 등의 다양한 용도에 쓰일 수 있음
- 결제 실패 처리
    - 결제 상태 추적
        - 결제 주기의 모든 단계에서 결제 상태를 정확하게 유지하는 것은 매우 중요함
        - 실패가 일어날 때마다 결제 거래의 현재 상태를 파악, 재시도 또는 환불이 필요한지 여부를 결정
        - 결제 상태는 데이터 추가만 가능한 데이터베이스 테이블에 보관
        - 재시도 큐 및 실패 메시지 큐
            - 재시도 큐: 일시적 오류 같은 재시도 가능 오류는 재시도 큐에 보냄
            - 실패 메시지 큐: 잘못된 입력 등으로 반복적으로 처리에 실패한 메시지는 결국에는 실패 메시지 큐로 보냄. 문제가 있는 메시지를 디버깅 및 격리하여 성공적으로 처리되지 않은 이유를 파악하기 위한 검사에 이용
        - 실패한 결제의 처리 흐름
            
            ![image.png](https://possible-telephone-196.notion.site/image/attachment%3A9ad66115-9db5-4e90-bf95-d21b6c90921d%3Aimage.png?table=block&id=335be4ca-d9d2-80b9-90d4-d7415637e506&spaceId=594418fe-2b8a-40ce-8d83-fb1b120f4b8f&width=710&userId=&cache=v2)
            
            1. 재시도 가능한지 확인
                1. 재시도 가능 실패 → 재시도 큐로 전송
                2. 재시도 불가능 실패 → 오류 내역을 데이터베이스에 저장
            2. 결제 시스템: 재시도 큐에 쌓인 이벤트를 읽어 실패한 결제를 재시도
            3. 결제 거래가 다시 실패하는 경우
                1. 재시도 횟수가 임계값 이내 → 다시 재시도 큐로
                2. 임계값 이상 → 실패 메시지 큐로
- 정확히 한 번 (exact-once) 전달
- 일관성
    - 결제 실행 과정에서 상태 정보를 유지 관리하는 여러 서비스가 호출 됨
        1. 결제 서비스는 비중복 난수, 토큰, 결제 주문, 실행 상태 등의 결제 관련 데이터를 유지 관리
        2. 원장은 모든 회계 데이터를 보관
        3. 지갑은 판매자의 계정 잔액을 유지
        4. PSP는 결제 실행상태를 유지
        5. 데이터는 안정성을 높이기 위해 여러 데이터 베이스 사본에 복제될 수 있음
    - 결제 시스템에 발생 가능한 데이터 일관성 문제를 해결하는 기법들
        - 내부 서비스 간의 데이터 일관성: 정확히 한 번 처리
        - 내부서비스 ↔ 외부서비스(PSP) 간의 일관성: 멱등성, 조정 프로세스
    - 데이터를 다중화하는 경우, 복제 지연으로 인해 기본 데이터베이스 ↔ 사본 데이터베이스의 불일치 발생 시 해결 방법
        1. 주 데이터베이스에서만 읽기와 쓰기 연산 처리
            - 설정은 쉽지만 규모 확장성이 떨어지는 단점
            - 사본은 안정성 보장에만 활용 & 트래픽은 처리 X → 자원이 낭비 됨
        2. 모든 사본이 항상 동기화 되도록 
            1. 팩서스(Paxos), 래프트(Raft) 등의 합의 알고리즘
            2. YugabyteDb, CocroachDB와 같은 합의 기반 분산 데이터베이스 사용
- 결제 보안 기술들
    
    
    | 위협 유형 | 예방책 |
    | --- | --- |
    | 요청/응답 도청 | HTTPS 사용  |
    | 데이터 변조 | 암호화 및 무결성 강화 모니터링 |
    | 중간자 공격 (MITM) | 인증서 고정(certificate pinning)과 함께 SSL 사용 |
    | 데이터 손실 | 여러 지역에 걸쳐 데이터베이스 복제 및 스냅숏 생성 |
    | DDoS 공격 | 처리율 제한 및 방화벽 |
    | PCI 규정 준수 | PCI DSS: 브랜드 신용 카드를 처리하는 조직을 위한 정보 보안 표준 |
    | 사기 (Fraud) | 주소 확인, 카드 확인번호(CVV), 사용자 행동 분석 등 |
    | 카드 도난 | 토큰화. 실제 카드 번호를 사용하는 대신 토큰을 저장하고 결제에 사용 |
    
    ## 4단계: 마무리
    
    언급할 가치가 있는 다른 주제들
    
    - 모니터링: 주요 지표를 모니터링하는 것을 통해 “특정 결제 수단의 평균 승인율은 얼마인가?”, “서버의 CPU 사용량은 얼마인가?” 같은 질문에 답을 얻을 수 있다. 대시보드도 생성 가능
    - 경보: 비정상적인 상황이 발생 시 온콜(on-call) 중인 개발자에게 알려 신속하게 대응할 수 있도록 하는 것이 중요
    - 디버깅 도구: 엔지니어와 고객 지원팀이 더 쉽게 디버깅 할 수 있도록 결제 거래의 상태, 처리 서버 기록, PSP 기록 등을 검토할 수 있는 도구를 개발
    - 환율: 국제적인 결제 시스템을 설계할 때의 고려사항
    - 지역: 지역마다 가용한 결제 수단이 완전히 달라질 수 있음
    - 현금 결제
    - 구글/애플 페이 연동
