## 📊 지표 모니터링 및 경보 시스템 설계

### 1️⃣ 시스템 요구사항

**규모**

- DAU 1억 명
- 서버 풀 1,000개 (각 100대) → 총 1,000만 개 지표 수집
- 데이터 보관: 1년 (7일→1분 단위→1시간 단위로 다운샘플링)

**비기능 요구사항**

- 규모 확장성, 낮은 응답 지연, 높은 안정성, 유연한 파이프라인

---

### 2️⃣ 핵심 컴포넌트 (5가지)

```
데이터 수집 → 데이터 전송 → 데이터 저장 → 경보 → 시각화
```

**데이터 모델**: 시계열 데이터

- 지표 이름 + 태그/레이블 + (값, 타임스탬프) 배열

---

### 3️⃣ 데이터 수집 모델 비교

| 구분        | 풀(Pull) 모델         | 푸시(Push) 모델             |
|-----------|--------------------|-------------------------|
| **방식**    | 수집기가 주기적으로 데이터 가져옴 | 서버가 직접 수집기에 전송          |
| **대표 사례** | Prometheus         | CloudWatch, Graphite    |
| **장점**    | 디버깅 용이, 상태 진단 쉬움   | 짧은 프로세스 처리, 복잡한 네트워크 대응 |
| **단점**    | 네트워크 구성 복잡         | 신뢰성 검증 필요               |

**핵심 인프라**

- 풀 모델: 서비스 탐색 서비스(SDS) + 안정 해시 링으로 중복 방지
- 푸시 모델: 로드밸런서 + Auto Scaling 클러스터

---

### 4️⃣ 규모 확장 전략

**카프카 활용**

```
수집기 → Kafka → 스트림 처리(Storm/Flink) → 시계열 DB
```

- 장점: 컴포넌트 분리, 장애 시 데이터 보존, 파티션 기반 확장
- 파티션 전략: 지표명, 태그, 우선순위별 분류

**저장소 최적화**

1. **데이터 압축**: 기준값 대비 차이만 저장
2. **다운샘플링**: 7일→1분→1시간 단위로 해상도 감소
3. **냉동 저장소**: 비활성 데이터 별도 보관

---

### 5️⃣ 주요 서비스 설계

**질의 서비스**

- 캐시 계층 도입으로 DB 부하 감소
- 시계열 DB 전용 질의어(Flux) 활용
- 클라이언트-DB 결합도 낮춤

**경보 시스템**

```
설정 파일(YAML) → 경보 관리자 → Kafka → 소비자 → 알림 발송
```

- 기능: 필터링, 중복 제거, 재시도, 접근 제어
- 저장소: 카산드라 등 키-값 DB로 상태 관리

**시각화**

- 상용 솔루션 활용 권장 (예: Grafana)

---

### 6️⃣ 시계열 DB 선택

**추천**: InfluxDB, Prometheus

- 메모리 캐시 + 디스크 저장소 혼합
- 시계열 데이터 최적화로 적은 서버로 대용량 처리
- 빠른 실시간 분석 지원

**주의사항**

- Label 값의 가짓수는 낮게 유지
- 최근 26시간 데이터가 질의의 85% → 최신 데이터 최적화 중요
