**설계의 트레이드오프, 면접 질문, 심화 토론**을 위주로 진행하기 위한 정리 자료입니다.

## 1. 핵심 요약 (발표 자료용 상세 버전)

### 1.1. 목표 및 요구사항 정의
*   **시스템 목표**: 10억 명의 사용자가 이용하는 대규모 이메일 서비스 구축.
*   **핵심 기능**:
    *   이메일 발송/수신 (SMTP, IMAP/POP 대신 HTTP API 활용 가정).
    *   이메일 목록 조회, 읽음 처리, 검색.
    *   스팸/바이러스 필터링.
*   **주요 비기능 요구사항**:
    *   **안정성(Reliability)**: 이메일 데이터 절대 소실 금지.
    *   **가용성(Availability)**: 데이터 복제 및 장애 격리.
    *   **확장성(Scalability)**: 사용자 및 데이터 증가에 따른 선형적 확장.

### 1.2. 개략적 규모 추정
*   **트래픽**:
    *   발송 QPS: 약 100,000 (10억 명 * 10건 / 10^5초).
    *   수신 QPS: 발송의 약 4배 (10억 명 * 40건).
*   **스토리지**:
    *   메타데이터: 1년에 약 730PB.
    *   첨부파일: 1년에 약 1,460PB.
    *   => **분산 데이터베이스 및 객체 저장소 필수**.

### 1.3. 아키텍처: 분산 시스템 접근
*   **전통적 메일 서버의 한계**: 파일 시스템(Maildir) 기반은 디스크 I/O 병목 및 검색/백업의 어려움 존재.
*   **분산 아키텍처 컴포넌트**:
    *   **웹 서버**: 기본적인 요청 처리, 인증, 초기 검증.
    *   **실시간 서버 (Stateful)**: WebSocket 등을 통해 새 메일 알림 푸시.
    *   **메타데이터 DB**: 이메일 헤더, 본문, 상태 정보 저장 (NoSQL/분산 DB).
    *   **첨부파일 저장소**: Amazon S3와 같은 객체 저장소.
    *   **분산 캐시 (Redis)**: 최근 이메일 조회 성능 향상.
    *   **검색 저장소**: Elasticsearch 등 역인덱스 기반 엔진.

    ![분산 메일 서버 아키텍처](https://drive.google.com/thumbnail?id=1ituqU61mFzrHEw1QBTr7w2KJ0RhgeAKj&sz=w500)

### 1.4. 이메일 전송/수신 절차
*   **전송 절차**:
    1.  사용자 요청 -> 로드밸런서 -> 웹 서버.
    2.  웹 서버: 검증 후 **메시지 큐(Kafka)**에 넣음. (첨부파일은 S3에 저장 후 참조만 큐에)
    3.  **SMTP 워커**: 큐에서 메시지 꺼내 스팸 검사 후 외부 SMTP 서버로 전송.
*   **수신 절차**:
    1.  SMTP 로드밸런서 -> SMTP 서버.
    2.  이메일을 **수신 큐**에 넣음.
    3.  **메일 처리 워커**: 스팸 필터링 후 메타데이터 DB/S3에 저장.
    4.  실시간 서버를 통해 사용자에게 알림.

    ![이메일 수신 절차](https://drive.google.com/thumbnail?id=1i0WUW-zQi3USDRAuz7LffjLSU_3LQkai&sz=w700)

### 1.5. 데이터 저장소 설계
*   **데이터 특성**:
    *   헤더는 작고 빈번하게 접근, 본문은 크고 한 번만 읽는 경향.
    *   **높은 안정성 필수**.
*   **DB 선정**:
    *   RDBMS: 검색엔 좋으나 대용량 BLOB 저장엔 부적합.
    *   **NoSQL (Cassandra/Bigtable)**: 대규모 확장성 및 쓰기 성능 우수.
*   **파티셔닝 전략**: `user_id`를 파티션 키로 사용하여 한 사용자의 데이터를 한 노드(또는 복제 그룹)에 모음.

### 1.6. 검색 기능 구현
*   **특징**: 쓰기(이메일 수신/전송)가 읽기(검색)보다 훨씬 빈번함. (Write-heavy)
*   **방안**:
    1.  **Elasticsearch**: 통합 쉽고 성능 좋으나, 주 데이터 저장소와 동기화(일관성) 문제 존재.
    2.  **LSM 트리 기반 자체 엔진**: 쓰기 성능 최적화(순차 쓰기). 디스크 I/O 병목 해결.

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)

### 2.1. 검색 엔진: Elasticsearch vs 자체 구축 (LSM Tree)
| 특징 | **Elasticsearch** | **자체 구축 (LSM Tree)** |
| :--- | :--- | :--- |
| **장점** | 강력한 텍스트 검색, 구축 용이, 풍부한 생태계 | 이메일 특화 최적화 가능, 단일 시스템으로 운영 가능 (복잡도 감소) |
| **단점** | 주 DB와 데이터 동기화 지연/불일치, 별도 운영 비용 | 개발 난이도 극상, 유지보수 어려움 |
| **적합성** | **일반적인 규모, 빠른 구축 필요 시** | **Gmail 같은 초대형 서비스, 극한의 최적화 필요 시** |

> **논의 포인트**:
> * "스타트업이라면?" (당연히 Elasticsearch)
> * "데이터 일관성이 깨지면(DB엔 있는데 검색 안 됨) 어떻게 복구하나?" (재색인 프로세스 필요)

### 2.2. 데이터 일관성 vs 가용성 (CAP 이론)
| 선택 | **일관성 (Consistency)** | **가용성 (Availability)** |
| :--- | :--- | :--- |
| **전략** | 주 사본(Primary)에만 쓰고 읽기. 장애 시 서비스 중단(CP). | 아무 사본에나 쓰고 읽기. 장애 시에도 서비스 가능(AP). |
| **이메일 서비스** | **선택함**. 이메일 유실/안 보임은 치명적이므로, 장애 시 차라리 접근을 막고 복구를 기다림. | (이메일에서는 잘 선택 안 함. 데이터 꼬이면 복구 힘듦) |

### 2.3. 프로토콜: SMTP/IMAP vs HTTP API
| 프로토콜 | **SMTP/IMAP/POP** | **HTTP API (REST/JSON)** |
| :--- | :--- | :--- |
| **표준성** | 전 세계 표준, 모든 메일 클라이언트 호환 | 서비스 전용, 웹/앱 개발 용이 |
| **효율성** | 바이너리 전송, 연결 유지 등 (상황 따라 다름) | 오버헤드 있음, 모바일 친화적 |
| **선택** | **서버 간 전송(MTA)**에는 필수 | **클라이언트-서버 통신**에는 HTTP가 대세 (Gmail API 등) |

## 3. 면접 예상 질문 (Interview Q&A)

### Q1. "이메일 전송 실패 시 어떻게 처리하나요?"
* **핵심 키워드**: `재시도 큐(Retry Queue)`, `지수적 백오프(Exponential Backoff)`, `데드 레터 큐(DLQ)`
* **답변 논리**:
    * 일시적 오류(네트워크 등)는 **지수적 백오프**를 적용하여 재시도합니다.
    * 영구적 오류(주소 불명 등)는 즉시 실패 처리하고 사용자에게 알립니다.
    * 재시도 횟수를 초과하면 **데드 레터 큐**로 보내 개발자가 분석하거나 별도 처리합니다.

### Q2. "스팸 메일은 어떻게 걸러내나요?"
* **핵심 키워드**: `IP 평판(Reputation)`, `SPF/DKIM/DMARC`, `머신러닝 분류기`
* **답변 논리**:
    * **전송 시**: 전용 IP 사용, IP 워밍업(서서히 전송량 증가), 인증 프로토콜(SPF 등) 준수.
    * **수신 시**: 블랙리스트 IP 차단, 콘텐츠 기반 머신러닝 필터링(Bayesian filter 등) 적용.

### Q3. "사용자별 데이터 파티셔닝 시 '핫스팟' 문제는 없나요?"
* **핵심 키워드**: `사용자별 격리`, `데이터 균등 분포`
* **답변 논리**:
    * 이메일 서비스는 SNS(연예인 팔로우)와 달리, 한 사용자의 메일함 데이터가 다른 사용자에게 영향을 주지 않습니다.
    * `user_id` 해시 기반 파티셔닝은 데이터를 매우 균등하게 분산시키므로, 특정 사용자에게 트래픽이 몰리는 핫스팟 문제는 거의 발생하지 않습니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)

### 주제 A: 이메일 보안과 암호화 (E2EE)
* **상황**: 서버 운영자조차 이메일 내용을 볼 수 없게 하려면?
* **접근법**:
    * **End-to-End Encryption (PGP 등)**: 클라이언트에서 암호화하여 전송. 검색 기능 구현이 매우 어려워짐(암호화된 텍스트 검색 불가).

### 주제 B: 대용량 첨부파일 처리 최적화 (De-duplication)
* **상황**: 100명이 같은 10MB 파일을 서로에게 전송한다면 스토리지 낭비가 심함.
* **접근법**:
    * **중복 제거(De-duplication)**: 파일의 해시값(SHA-256)을 계산하여, 이미 저장소에 있는 파일이면 저장하지 않고 참조 카운트만 올림.

### 주제 C: 메일 검색의 실시간성 확보
* **상황**: 메일 수신 직후 검색이 안 되면 사용자 불만 발생.
* **접근법**:
    * **Near Real-time Indexing**: Kafka 컨슈머가 메타데이터 DB 저장과 동시에(또는 직후) 검색 엔진에 인덱싱 요청.
    * 검색 엔진의 Refresh Interval 조정.

## 5. 추가 학습 추천
* **LSM Tree (Log-Structured Merge-tree)**: 쓰기 중심 DB(Cassandra, RocksDB)의 핵심 자료구조.
* **Email Auth Protocols**: SPF, DKIM, DMARC의 작동 원리 (보안 필수 지식).
* **Bloom Filter**: 이메일 수신 시 악성 도메인/IP를 빠르게 필터링하는 확률적 자료구조.
