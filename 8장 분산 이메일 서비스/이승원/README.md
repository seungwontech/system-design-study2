# 분산 이메일 서비스 설계

이번 글에서는 **지메일(Gmail), 아웃룩(Outlook), 야후 메일(Yahoo Mail)** 과 같은 대규모 이메일 서비스를 어떻게 설계할 수 있는지 살펴본다. 이메일 시스템은 수억 명의 사용자가 사용하는
서비스이기 때문에 단순한 메일 서버가 아니라 **확장성과 안정성을 고려한 분산 시스템** 으로 설계되어야 한다.

---

# 1. 문제 이해 및 설계 범위

대규모 이메일 시스템을 설계하기 위해 먼저 요구사항을 정의해야 한다.

가정하는 사용자 규모는 **약 10억 명**이다. 사용자는 이메일을 보내고 받을 수 있어야 하며, 받은 메일을 조회하거나 읽음 여부에 따라 필터링할 수 있어야 한다. 또한 제목, 발신자, 본문 내용 등을 기반으로 *
*검색 기능**을 제공해야 하며, 스팸 및 바이러스 필터링 기능도 필요하다.

사용자는 이메일 서버에 접속하기 위해 전통적으로 **SMTP, POP, IMAP** 과 같은 프로토콜을 사용한다. 이러한 프로토콜은 오래된 방식이지만 여전히 널리 사용되고 있다. 그러나 웹메일 환경에서는 **HTTP
기반 API**를 이용해 이메일 시스템과 통신하기도 한다.

또한 현대 이메일 서비스에서는 **첨부 파일 기능** 역시 중요한 요소이다.

---

# 2. 비기능 요구사항

대규모 이메일 시스템은 기능뿐만 아니라 다음과 같은 비기능 요구사항도 충족해야 한다.

첫째, **데이터 안정성**이다. 이메일 데이터는 사용자에게 매우 중요한 정보이기 때문에 절대 유실되어서는 안 된다.

둘째, **고가용성**이다. 이메일과 사용자 데이터를 여러 서버에 자동으로 복제하여 일부 서버에 장애가 발생하더라도 시스템이 계속 동작해야 한다.

셋째, **확장성**이다. 사용자 수와 이메일 수가 계속 증가하더라도 시스템 성능이 저하되지 않아야 한다.

넷째, **유연성**이다. 새로운 기능을 쉽게 추가할 수 있도록 시스템 구조가 확장 가능하게 설계되어야 한다.

또한 기존 이메일 프로토콜인 **POP, IMAP**은 기능이 제한적이기 때문에, 필요에 따라 서비스 전용 프로토콜을 설계할 수도 있다.

---

# 3. 개략적인 규모 추정

대규모 이메일 시스템에서는 시스템이 처리해야 할 트래픽과 데이터 규모를 추정하는 것이 중요하다.

사용자가 **10억 명**이라고 가정하고 한 사용자가 하루 평균 **10개의 이메일을 보낸다**고 하면 이메일 전송 QPS는 약 **100,000** 정도가 된다.

또한 한 사용자가 하루 평균 **40개의 이메일을 수신**한다고 가정하면 메타데이터 저장량도 매우 커진다.

이메일 메타데이터 크기를 평균 **50KB**라고 가정하면 1년 동안 저장해야 할 메타데이터의 양은 다음과 같다.

```
10억 사용자 × 하루 40개 × 365일 × 50KB
≈ 750PB
```

여기에 첨부 파일까지 고려하면 저장 공간은 더욱 증가한다.

첨부 파일이 포함된 이메일의 비율을 **20%**, 첨부 파일 평균 크기를 **500KB**라고 가정하면 1년 동안 필요한 첨부 파일 저장 공간은 약 **1460PB**에 달한다.

이처럼 이메일 서비스는 **엄청난 규모의 데이터 저장 시스템**이 필요하다.

---

# 4. 이메일 프로토콜 이해

이메일 시스템을 이해하기 위해서는 이메일 전송과 수신에 사용되는 주요 프로토콜을 알아야 한다.

## SMTP

SMTP(Simple Mail Transfer Protocol)는 이메일을 **한 메일 서버에서 다른 메일 서버로 전송할 때 사용하는 프로토콜**이다. 즉, 이메일을 보내는 역할을 담당한다.

---

## POP

POP(Post Office Protocol)은 메일 서버에 저장된 이메일을 **클라이언트로 다운로드하는 프로토콜**이다.

POP 방식에서는 이메일을 다운로드하면 서버에서 삭제되는 경우가 많기 때문에 **한 기기에서만 이메일을 확인하는 방식**에 적합하다.

---

## IMAP

IMAP(Internet Mail Access Protocol)은 이메일을 서버에 저장한 상태로 두고 필요할 때 조회하는 방식의 프로토콜이다.

이 방식에서는 이메일이 서버에 남아 있기 때문에 **여러 기기에서 동일한 이메일을 확인할 수 있다**. 또한 이메일을 열기 전에 헤더만 먼저 가져오기 때문에 비교적 효율적인 방식이다.

---

## HTTPS

HTTPS는 이메일 전송 프로토콜은 아니지만 **웹 기반 이메일 서비스(Webmail)**에서 클라이언트와 서버 간 통신을 위해 사용된다.

예를 들어 Outlook의 ActiveSync와 같은 서비스는 HTTPS 기반 통신을 이용한다.

---

## DNS와 MX 레코드

이메일을 전송할 때는 수신자의 메일 서버를 알아야 한다. 이를 위해 **DNS의 MX(Mail Exchange) 레코드**를 조회한다.

예를 들어 `gmail.com`으로 이메일을 보내려고 하면 DNS에서 gmail의 MX 레코드를 조회하여 **메일 서버 목록과 우선순위**를 확인한다.

메일 서버는 **우선순위 값이 낮은 서버부터 연결을 시도**하며, 연결에 실패하면 다음 우선순위 서버로 전송을 시도한다.

---

# 5. 첨부 파일 처리 방식

이메일 시스템에서 첨부 파일은 중요한 기능 중 하나이다. 이메일은 기본적으로 텍스트 기반 프로토콜이기 때문에 이미지, 영상, 문서와 같은 바이너리 데이터를 그대로 전송할 수 없다. 따라서 첨부 파일은 일반적으로 *
*Base64 인코딩 방식**을 사용하여 전송된다.

또한 이메일 첨부 파일은 대부분 크기 제한이 존재한다. 예를 들어 Gmail은 약 **25MB**, Outlook은 약 **20MB** 정도의 첨부 파일 제한을 가진다.

첨부 파일을 전송하기 위해서는 **MIME(Multipurpose Internet Mail Extension)** 이라는 표준이 사용된다. MIME은 이메일 메시지에 첨부 파일과 다양한 멀티미디어 데이터를 포함할 수
있도록 하는 규격이다.

---

# 6. 전통적인 이메일 서버 아키텍처

이메일 시스템의 기본 동작을 이해하기 위해 먼저 **전통적인 이메일 서버 구조**를 살펴볼 필요가 있다.

일반적인 이메일 전송 과정은 다음과 같은 단계로 이루어진다.

1. 사용자가 메일 클라이언트에서 이메일을 작성하고 전송 버튼을 누른다.
2. 이메일 클라이언트는 **SMTP 프로토콜**을 사용하여 발신 메일 서버로 메시지를 전달한다.
3. 발신 메일 서버는 **DNS에서 MX 레코드를 조회**하여 수신자의 메일 서버를 찾는다.
4. 발신 메일 서버는 수신 메일 서버로 이메일을 전달한다.
5. 수신 메일 서버는 이메일을 저장소에 저장한다.
6. 사용자가 메일 클라이언트에 로그인하면 **IMAP 또는 POP 프로토콜**을 사용하여 이메일을 가져온다.

이 과정은 비교적 단순하지만, 사용자 수가 매우 많아지면 단일 서버 구조로는 시스템을 운영하기 어렵다.

---

# 7. 이메일 저장 방식

전통적인 메일 서버는 이메일을 **파일 시스템에 저장**하는 방식으로 동작한다. 대표적인 방식이 **Maildir 구조**이다.

Maildir 구조에서는 각 사용자에게 독립적인 디렉터리가 생성되며 이메일은 개별 파일로 저장된다.

일반적인 Maildir 구조는 다음과 같다.

```
/home
 ├ user1
 │   └ Maildir
 │      ├ new
 │      ├ cur
 │      └ tmp
 └ user2
     └ Maildir
        ├ new
        ├ cur
        └ tmp
```

각 디렉터리의 의미는 다음과 같다.

* **new** : 새로 도착한 이메일
* **cur** : 사용자가 확인한 이메일
* **tmp** : 이메일 저장 중 임시 공간

이 구조는 단순하고 안정적이지만, 대규모 시스템에서는 여러 가지 한계가 존재한다.

---

# 8. 전통적인 이메일 시스템의 한계

전통적인 이메일 시스템은 다음과 같은 문제점을 가진다.

첫째, **확장성 문제**이다. 사용자 수가 수억 명으로 증가하면 단일 서버 기반 구조로는 트래픽을 처리하기 어렵다.

둘째, **디스크 I/O 병목 문제**이다. 이메일과 첨부 파일이 파일 시스템에 저장되기 때문에 디스크 접근이 빈번하게 발생한다.

셋째, **가용성 문제**이다. 서버 장애가 발생하면 이메일 서비스 전체가 영향을 받을 수 있다.

넷째, **검색 기능의 어려움**이다. 이메일 내용을 검색하기 위해서는 별도의 검색 인덱싱 시스템이 필요하다.

이러한 문제를 해결하기 위해 대규모 이메일 서비스는 **분산 아키텍처**를 기반으로 설계된다.

---

# 9. 분산 이메일 서버 아키텍처

대규모 이메일 서비스는 다음과 같은 주요 구성 요소로 이루어진다.

### 웹메일(Webmail)

사용자가 브라우저를 통해 이메일을 보내고 받을 수 있는 인터페이스이다.

### 웹 서버

웹 서버는 다음과 같은 역할을 담당한다.

* 사용자 인증
* 이메일 API 처리
* 이메일 발송 요청 처리
* 메일 목록 조회

웹메일에서 발생하는 대부분의 요청은 **HTTP 기반 API**를 통해 웹 서버로 전달된다.

### 실시간 서버

실시간 서버는 새로운 이메일이 도착했을 때 클라이언트에게 **즉시 알림을 전달하는 역할**을 한다.

실시간 통신 방식으로는 다음과 같은 기술이 사용될 수 있다.

* Long Polling
* Server-Sent Events
* WebSocket

대규모 시스템에서는 **WebSocket 방식**이 많이 사용된다.

---

# 10. 저장 계층(Storage Layer)

이메일 시스템에서는 다양한 종류의 데이터를 저장해야 하기 때문에 여러 종류의 저장소가 사용된다.

### 메타데이터 데이터베이스

메일 제목, 발신자, 수신자, 본문 등 이메일의 주요 정보를 저장한다.

### 첨부 파일 저장소

첨부 파일은 일반적으로 **Amazon S3와 같은 Object Storage**에 저장된다.
대용량 데이터를 효율적으로 저장하고 확장하기 위해서이다.

### 캐시 시스템

최근 이메일 데이터를 캐시에 저장하여 조회 성능을 높인다.
보통 **Redis** 같은 인메모리 캐시가 사용된다.

### 검색 저장소

이메일 검색 기능을 지원하기 위해 **역 인덱스(inverted index)** 기반 검색 시스템을 사용한다.
대표적인 예로 **Elasticsearch**가 있다.

---

# 11. 이메일 전송 절차

분산 시스템에서 이메일 전송 과정은 다음과 같이 이루어진다.

1. 사용자가 웹메일에서 이메일을 작성하고 전송 요청을 보낸다.
2. 요청은 **로드밸런서**를 거쳐 웹 서버로 전달된다.
3. 웹 서버는 이메일 형식, 첨부 파일 크기 등을 검증한다.
4. 검증이 완료되면 메시지는 **메시지 큐**에 저장된다.
5. SMTP 작업 프로세스가 메시지 큐에서 이메일을 가져온다.
6. 이메일은 발신 메일 서버에 저장된다.
7. SMTP 프로세스는 수신자의 메일 서버로 이메일을 전송한다.

---

# 12. 이메일 수신 절차

이메일이 외부 메일 서버에서 도착했을 때 시스템 내부에서는 여러 단계의 처리 과정을 거친다. 대규모 이메일 시스템에서는 단순히 메시지를 저장하는 것이 아니라 **보안 검사, 스팸 필터링, 저장 처리, 실시간 알림
전달** 등의 과정이 함께 수행된다.

일반적인 이메일 수신 과정은 다음과 같다.

1. 외부에서 전송된 이메일이 **SMTP 로드밸런서**에 도착한다.
2. 로드밸런서는 트래픽을 여러 **SMTP 서버**로 분산한다.
3. SMTP 서버는 기본적인 이메일 정책을 검증한다.
4. 이메일 메시지는 **수신 이메일 큐**에 저장된다.
5. 이메일 처리 작업 프로세스(worker)가 큐에서 메시지를 가져와 스팸 및 바이러스 검사를 수행한다.
6. 검사가 완료된 이메일은 **메타데이터 DB, 첨부 파일 저장소, 캐시** 등에 저장된다.
7. 수신자가 온라인 상태라면 **실시간 서버(WebSocket 등)** 를 통해 새 이메일 알림을 전달한다.
8. 사용자가 오프라인 상태라면 이메일은 저장소에 보관된다.
9. 사용자가 웹메일에 접속하면 웹 서버가 저장 계층에서 이메일 데이터를 조회하여 클라이언트로 전달한다.

이 과정에서 **수신 이메일 큐는 시스템 확장성을 높이는 핵심 구성 요소**이다. 큐를 통해 SMTP 서버와 이메일 처리 작업 프로세스 간의 결합도를 낮출 수 있으며, 각 컴포넌트를 독립적으로 확장할 수 있다.

# 13. 이메일 메타데이터 데이터베이스 설계

이메일 시스템에서 저장해야 하는 데이터는 크게 두 가지로 나뉜다.

* 이메일 **메타데이터**
* 이메일 **첨부 파일**

이메일 메타데이터는 다음과 같은 특징을 가진다.

* 일반적으로 크기가 작다.
* 읽기 요청이 매우 빈번하다.
* 대부분의 이메일은 한 번만 읽힌다.
* 최근 이메일 조회 요청이 많다.

이러한 특성을 고려하면 **고성능 읽기 처리와 확장성이 가능한 데이터베이스 구조**가 필요하다.

대규모 이메일 서비스에서는 보통 **NoSQL 기반 분산 데이터베이스**를 사용한다.

대표적인 예로는 다음과 같은 시스템이 있다.

* Cassandra
* Bigtable
* DynamoDB

---

# 14. 데이터 모델 설계

이메일 시스템의 데이터 모델을 설계할 때 가장 중요한 요소는 **데이터 분산 전략**이다.

일반적으로 이메일 데이터는 **user_id를 파티션 키(partition key)** 로 사용하여 분산 저장된다.

데이터 모델은 다음과 같은 구조로 설계할 수 있다.

### 사용자 폴더 테이블

```
folders_by_user

user_id
folder_id
folder_name
```

이 테이블은 특정 사용자의 모든 이메일 폴더 목록을 조회하는 데 사용된다.

---

### 폴더별 이메일 목록

```
emails_by_folder

user_id
folder_id
email_id
from
subject
preview
is_read
```

이 테이블은 특정 폴더에 포함된 이메일 목록을 조회하는 데 사용된다.

---

### 이메일 상세 정보

```
emails_by_user

user_id
email_id
from
to
subject
body
attachments
```

이 테이블은 이메일의 상세 내용을 저장한다.

---

### 첨부 파일 테이블

```
attachments

email_id
filename
url
```

첨부 파일 자체는 일반적으로 **Object Storage(S3)** 에 저장되고, 데이터베이스에는 파일 위치 정보만 저장된다.

---

# 15. 읽지 않은 이메일 처리 전략

이메일 시스템에서 매우 자주 발생하는 요청 중 하나는 **읽지 않은 이메일 조회**이다.

하지만 NoSQL 데이터베이스에서는 `is_read` 조건으로 필터링하는 것이 비효율적일 수 있다.

이 문제를 해결하기 위해 이메일 데이터를 다음과 같이 **비정규화(denormalization)** 할 수 있다.

### 읽은 이메일 테이블

```
read_emails

user_id
folder_id
email_id
from
subject
preview
```

### 읽지 않은 이메일 테이블

```
unread_emails

user_id
folder_id
email_id
from
subject
preview
```

이메일을 읽으면 해당 레코드를 `unread_emails`에서 삭제하고 `read_emails` 테이블로 이동시킨다.

이 방식은 데이터 중복이 발생하지만 **대규모 시스템에서 조회 성능을 크게 개선할 수 있다.**

---

# 16. 이메일 스레드(Thread) 관리

이메일 클라이언트는 보통 **대화형(thread) 보기** 기능을 제공한다. 이를 위해 이메일 메시지 헤더에는 다음과 같은 필드가 포함된다.

```
Message-ID
In-Reply-To
References
```

* **Message-ID** : 이메일 메시지의 고유 식별자
* **In-Reply-To** : 어떤 이메일에 대한 답장인지 나타냄
* **References** : 대화에 포함된 메시지 ID 목록

이 정보를 이용하면 이메일 클라이언트는 전체 대화 흐름을 구성할 수 있다.

---

# 17. 이메일 전송 가능성(Deliverability)

이메일 시스템에서 단순히 이메일을 보내는 것만으로는 충분하지 않다. 이메일이 실제로 수신자의 받은 편지함에 도달해야 한다.

이를 위해 다음과 같은 요소가 중요하다.

### 전용 IP 사용

메일 서버는 **전용 IP 주소**를 사용해야 한다.
공유 IP는 스팸으로 분류될 가능성이 높다.

### 발신자 평판 관리

메일 서버의 IP 평판(reputation)이 좋지 않으면 이메일이 스팸으로 분류될 수 있다.

### 스팸 방지 인증

대표적인 이메일 인증 기술은 다음과 같다.

* **SPF (Sender Policy Framework)**
* **DKIM (DomainKeys Identified Mail)**
* **DMARC (Domain-based Message Authentication Reporting and Conformance)**

이 기술들은 이메일 발신자가 실제 도메인의 소유자인지 검증하는 역할을 한다.

---

# 18. 이메일 검색 시스템

사용자는 이메일을 제목, 발신자, 내용 등을 기준으로 검색할 수 있어야 한다.

검색 기능을 구현하는 방법은 크게 두 가지가 있다.

---

## 1️⃣ Elasticsearch 기반 검색

이메일 내용과 메타데이터를 **Elasticsearch**에 인덱싱하여 검색 기능을 제공한다.

이 방식의 특징은 다음과 같다.

* 빠른 텍스트 검색
* 높은 확장성
* 복잡한 검색 기능 지원

이메일 전송 및 수신 이벤트가 발생할 때마다 검색 인덱스를 **비동기 방식으로 업데이트**한다.

---

## 2️⃣ 맞춤형 검색 엔진

대규모 이메일 서비스에서는 자체 검색 엔진을 개발하기도 한다.

검색 엔진은 일반적으로 **LSM Tree(Log Structured Merge Tree)** 기반 저장 구조를 사용한다.

대표적인 예

* Cassandra
* RocksDB

LSM Tree는 **순차 쓰기(sequential write)** 를 기반으로 설계되어 대규모 데이터 처리에 적합하다.

---

# 19. 대규모 확장성과 가용성

이메일 서비스는 전 세계 사용자에게 제공되기 때문에 **멀티 데이터센터 구조**로 설계된다.

이 구조의 특징은 다음과 같다.

* 여러 데이터센터에 데이터 복제
* 장애 발생 시 다른 데이터센터로 자동 전환
* 사용자와 가까운 서버로 트래픽 라우팅

이를 통해 시스템은 높은 가용성을 유지할 수 있다.

---

# 20. 결함 내성 (Fault Tolerance)

대규모 이메일 시스템에서는 서버 장애, 네트워크 문제, 메시지 처리 지연 등 다양한 장애 상황이 발생할 수 있다. 따라서 일부 컴포넌트에 문제가 생기더라도 시스템이 계속 동작하도록 설계해야 한다.

이를 위해 이메일 시스템은 **데이터 복제(replication)**, **메시지 큐 기반 비동기 처리**, **로드밸런서를 통한 장애 전환(failover)** 등의 방식을 사용한다.

---

# 21. 규정 준수 (Compliance)

이메일 서비스는 전 세계 사용자 데이터를 처리하기 때문에 각 국가의 법적 규제를 준수해야 한다.

예를 들어 유럽에서는 **GDPR(General Data Protection Regulation)**에 따라 개인정보 보호와 데이터 삭제 요청 처리 등의 요구사항을 충족해야 한다. 또한 일부 국가에서는 법적 절차에
따른 **합법적 감청(legal intercept)** 기능이 요구될 수 있다.

---

# 22. 보안 (Security)

이메일에는 민감한 정보가 포함될 수 있기 때문에 다양한 보안 기능이 필요하다.

대표적인 보안 기능은 다음과 같다.

* 피싱 이메일 탐지
* 악성 링크 및 첨부 파일 검사
* 의심스러운 로그인 시도 차단
* TLS 기반 이메일 전송 암호화

또한 이메일 인증 기술인 **SPF, DKIM, DMARC**를 통해 발신자의 신뢰성을 검증한다.

---

# 23. 최적화 (Optimization)

대규모 이메일 시스템에서는 저장 공간과 성능을 최적화하는 것도 중요하다.

예를 들어 동일한 첨부 파일이 여러 사용자에게 전송되는 경우 **첨부 파일 중복 제거(deduplication)** 기술을 사용하여 저장 공간을 절약할 수 있다. 이를 위해 파일의 해시 값을 이용해 동일한 파일
여부를 확인한다.

---

# 마무리

대규모 이메일 시스템은 단순한 메일 서버가 아니라 다음과 같은 여러 구성 요소가 결합된 복잡한 분산 시스템이다.

- SMTP 기반 이메일 전송

- IMAP/POP 기반 이메일 수신

- 메시지 큐 기반 비동기 처리

- 분산 데이터베이스 저장 구조

- 첨부 파일 Object Storage

- 검색 인덱스 시스템

- 스팸 및 보안 인증 체계

- 글로벌 데이터센터 기반 확장성

이러한 설계를 통해 이메일 서비스는 수십억 명의 사용자를 안정적으로 지원할 수 있다.
