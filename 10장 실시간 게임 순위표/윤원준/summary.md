# 10장: 실시간 게임 순위표 핵심 요약 및 심화 분석

## 1. 핵심 요약
### 시스템 이해 및 설계 요구사항
*   **목적:** 매달 초기화되는 실시간 게임 순위표 설계. 승리 시 점수 1점씩 증가.
*   **주요 기능:** 상위 10명의 플레이어 표시, 특정 사용자의 순위 표시, 특정 사용자 기준 위/아래 4명 표시.
*   **규모 추정:** DAU 500만, MAU 2500만. 1인당 하루 평균 10게임 플레이. 점수 갱신 QPS 평균 500, 최대 2500. 순위표 조회 QPS 약 50.

### 아키텍처 및 데이터 흐름
*   **점수 갱신 주체:** 보안 상(중간자 공격 방지) 클라이언트가 아닌 **게임 서버**가 순위표 서비스에 점수 갱신을 요청해야 함.
*   **개략적 설계:** 클라이언트 -> 게임 서비스 -> 순위표 서비스 -> 순위표 저장소.
*   필요에 따라 게임 서버와 다른 서비스들(순위표, 분석, 푸시 알림) 사이에 Kafka와 같은 메시지 큐(MQ)를 두어 결합도를 낮출 수 있음.

### 핵심 기술: Redis 정렬 집합 (Sorted Set)
*   **RDBMS의 한계:** 수백만 건의 데이터를 실시간으로 정렬하여 순위를 매기기엔 너무 느림. (인덱싱을 해도 특정 사용자 순위를 바로 찾기 어려움)
*   **Redis 채택:** 인메모리 기반으로 실시간성을 보장. 특히 `Sorted Set`은 내부적으로 **해시 테이블**과 **스킵 리스트(Skip List)**로 구현되어 있어 O(log N)의 시간 복잡도로 삽입/검색/순위 조회가 가능함.
    *   `ZADD`, `ZINCRBY`: 점수 추가/갱신
    *   `ZRANGE` / `ZREVRANGE`: 범위 내 사용자 조회
    *   `ZRANK` / `ZREVRANK`: 특정 사용자의 등수 조회
*   **저장소 요구사항:** MAU 2500만 명 기준 약 650MB(오버헤드 포함 최대 1.3GB 수준)로 단일 Redis 서버로도 충분히 감당 가능.

### 규모 확장성 (Scale-out) 방안
DAU가 5억 명 수준으로 100배 증가한다면 단일 Redis로는 불가능하므로 **샤딩(Sharding)**이 필요함.
*   **고정 파티션 (Fixed Partitioning):** 점수 구간대별로 샤드를 나눔(예: 1~100점 1번 샤드, 101~200점 2번 샤드). 특정 등수나 상위권 조회가 빠르지만, 점수가 한 구간에 몰리면 핫스팟이 발생하므로 동적 범위 조정이 필요함.
*   **해시 파티션 (Hash Partitioning):** 사용자의 ID를 해싱하여 분산. 데이터는 고르게 퍼지지만 "상위 10명"을 구하려면 모든 샤드에 질의한 뒤 취합(Scatter-Gather)하고 재정렬해야 하므로 지연 시간 증가. (따라서 **순위표에는 고정 파티션이 더 적합**)

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)

*   **RDBMS vs Redis (Sorted Set)**
    *   *RDBMS:* 구현이 간단하고 데이터 영속성이 뛰어나나, `ORDER BY`를 통한 잦은 랭킹 쿼리는 CPU/디스크 I/O를 고갈시키며 실시간 처리에 취약.
    *   *Redis:* 메모리 기반으로 속도가 매우 빠르고 스킵 리스트 구조상 순위 조작이 O(log N)으로 최적화됨. 단, 데이터 영속성을 위해 백업(RDB/AOF)과 읽기 사본, 페일오버 구성이 필수적임.
*   **Redis Sharding: 고정 파티션 vs 해시 파티션**
    *   *해시 파티션:* 데이터와 부하 분산이 가장 완벽하지만, 전체 순위 범위 검색을 위해 *Scatter-Gather(분산-수집)* 패턴을 강제하므로 지연 시간이 크다.
    *   *고정 파티션:* 점수 대역으로 분리하므로 순위 기반 쿼리가 빠르고 한 파티션 내에서 처리가 가능함. 그러나 특정 점수대에 사용자가 몰릴 경우 핫 파티션 문제가 발생하여 파티션 임계치를 지속적으로 튜닝해야 함.
*   **Redis vs DynamoDB (NoSQL 대안)**
    *   NoSQL(DynamoDB 등)을 사용할 경우, 파티션 키(게임+달)와 정렬 키(점수)를 활용해 글로벌 보조 인덱스(GSI)로 랭킹을 구현할 수 있다. 관리형이라는 장점이 있지만, 핫 파티션을 막기 위해 '쓰기 샤딩(Write Sharding)'을 도입하면 결국 읽어올 때 전체 파티션을 병렬 조회해야 하므로 복잡성이 크게 증가함. 순위표 특화 자료구조를 가진 Redis가 일반적으로 우세.

## 3. 면접 예상 질문 (Interview Q&A)

**Q. 실시간 게임 순위표에서 관계형 데이터베이스(RDBMS)를 사용하지 않고 Redis를 선택한 이유는 무엇입니까?**
A. 실시간 순위표는 읽기/쓰기 빈도가 매우 높고, 특정 사용자의 현재 순위나 내 앞뒤 순위를 빠르게 알아내야 합니다. RDBMS에서 수백만 건의 점수를 빈번하게 `ORDER BY`로 정렬하고 `COUNT`로 등수를 구하는 것은 실시간 처리에 부적합합니다. 반면 Redis의 정렬 집합(Sorted Set)은 스킵 리스트 구조를 기반으로 데이터 삽입/갱신 및 순위 검색을 O(log N)의 일관된 성능으로 제공하기 때문에 선택했습니다.

**Q. 트래픽과 데이터가 급증해 단일 Redis 서버 메모리를 초과하게 된다면 어떻게 확장하시겠습니까?**
A. 데이터 샤딩(Sharding)을 도입하겠습니다. 사용자 ID 기반의 해시 파티셔닝은 데이터는 고르게 분산되지만 상위 N명을 조회하기 위해 모든 샤드를 찔러야 하는 성능 문제가 있습니다. 따라서 점수 범위를 기준으로 파티셔닝하는 **고정 파티션 방식**을 우선적으로 고려하겠습니다. 만약 특정 점수대에 유저가 집중된다면 범위 임계값을 동적으로 조정하거나 혼합 방식을 설계할 것입니다.

**Q. Redis 서버에 장애가 발생해 메모리 데이터가 소실되었다면, 순위표 시스템은 어떻게 복구해야 합니까?**
A. 1차적으로는 Redis 주 서버에 장애가 생기면 즉시 읽기 사본(Replica)을 주 서버로 승격시켜 가용성을 유지합니다. 만약 클러스터 전체 장애로 데이터가 유실되었다면, MySQL과 같은 영구 저장소(원본 DB나 이벤트 소싱 로그/Kafka 로그)에 저장된 승리 이력 데이터(이벤트)를 다시 읽어들여 사용자별로 `ZINCRBY`를 일괄 실행하는 오프라인 복구 스크립트를 돌려 재구축해야 합니다.

**Q. 점수가 동일한 동점자가 발생했을 때, 먼저 점수를 달성한 사람이 더 높은 순위를 갖도록 하려면 어떻게 설계해야 하나요?**
A. Redis Sorted Set의 점수(Double 타입)를 세밀하게 활용할 수 있습니다. 기본 점수를 정수부로 사용하고, 점수 달성 시점의 타임스탬프를 소수부로 활용하여 소수부 값이 작을수록(시간이 빠를수록) 전체 점수가 높게 계산되도록 수학적으로 변환하여 저장하는 방법이 있습니다. 또는, Redis 해시에 사용자 ID와 마지막 득점 시간을 저장하여 동점 시 애플리케이션 레벨에서 비교하는 로직을 추가할 수도 있습니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)

*   **스킵 리스트(Skip List) 내부 원리:** B-Tree 등 일반 DB 인덱스 구조와 비교했을 때 스킵 리스트가 삽입/삭제 시 락(Lock) 경합 및 재구조화 오버헤드가 적은 이유와 이것이 인메모리 순위표 구현에 최적화된 배경.
*   **동점자 처리 고도화 기법 (Tie-breaker):** `(Score) + (Timestamp 역순 값)` 기법 외에, 여러 정렬 기준(예: 점수 -> 승률 -> 플레이 횟수)을 소수점으로 압축해 Sorted Set의 Score 1개 필드에 매핑하는 한계와 비트 연산(Bitwise) 트릭.
*   **무결성 보장 구조:** Kafka를 이용한 비동기 점수 업데이트 시, 재시도(Retry)로 인해 점수가 중복해서 1점씩 오르는 문제(At-least-once)를 방지하기 위한 멱등성(Idempotency) 보장 아키텍처 (예: `Event ID` 단위 체크 캐시 활용 등).

## 5. 추가 학습 추천
*   **Redis Sorted Set Internals:** Redis 공식 문서 또는 소스 코드(dict.c, t_zset.c)에서 스킵 리스트 구현 방식 학습.
*   **Amazon ElastiCache for Redis & DynamoDB GSI:** AWS 기반에서 관리형 Redis 클러스터링 구성 방식과 DynamoDB에서 글로벌 보조 인덱스를 사용한 랭킹 시스템 예제 비교 실습.
*   **Kafka Exactly-Once Semantics (EOS):** 스트리밍 환경에서 데이터 정확성(중복 합산 방지)을 보장하는 원리.
