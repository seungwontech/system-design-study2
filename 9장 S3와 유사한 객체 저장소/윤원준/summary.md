# 9장: S3와 유사한 객체 저장소 핵심 요약 및 심화 분석

## 1. 핵심 요약
### 시스템 이해
*   **객체 저장소의 특성:** 블록 저장소(가장 유연함, DB 등에 사용)나 파일 저장소(디렉터리 구조)와 달리, 데이터 영속성과 대규모 확장성, 낮은 비용에 초점을 맞춘 저장소. 계층 구조 없이 평면적인 형태로 객체(데이터+메타데이터)를 보관하며 RESTful API로 접근. (예: AWS S3)
*   **용어:** 버킷(전역 유일한 논리적 컨테이너), 객체(저장되는 개별 데이터, 불변성), 버전(복구를 위한 객체 이력 관리), URI(객체 식별자), SLA(99.999999999% 내구성, 99.9% 가용성 보장)

![저장소 유형](https://drive.google.com/thumbnail?id=1Zrqk8rogf4_w6FjW7hi5QDcv-3gYs_G-&sz=w700)

### 설계 요구사항
*   **기능:** 버킷 생성, 객체 업로드/다운로드, 버전 관리, 버킷 내 객체 목록 출력(ls).
*   **비기능:** 100PB 데이터, 높은 내구성(99.9999%), 높은 가용성(99.99%), 저장소 비용 최적화.

### 아키텍처 및 개략적 설계
*   **설계 철학:** UNIX 파일 시스템(inode와 파일 데이터 분리)과 유사하게 메타데이터(변경 가능, DB 사용)와 객체 데이터(불변, 디스크 저장)를 분리.
*   **컴포넌트:**
    *   **로드밸런서:** API 서비스로 요청 분산.
    *   **API 서비스:** 무상태 서버로 IAM, 메타데이터, 저장소 서비스 조율.
    *   **IAM 서비스:** 인증 및 권한 부여.
    *   **메타데이터 저장소:** 객체 이름, 버킷 정보, UUID 매핑 등을 보관.
    *   **데이터 저장소:** 실제 데이터를 보관하고 UUID로 접근.

![개략적 설계안](https://drive.google.com/thumbnail?id=1Iq4d5YCvMa9dUtKveIAUwpu7YoCVPpkT&sz=w700)

### 상세 설계 및 중요 메커니즘
*   **데이터 저장소 내부 구조:**
    *   **데이터 라우팅 서비스:** 무상태 서비스로 최적의 데이터 노드 결정, 데이터 읽기/쓰기 중계.
    *   **배치 서비스:** 데이터 노드 상태 모니터링(박동 메시지), 가상 클러스터 지도 유지. 데이터 사본 위치 결정 (장애 도메인 분리).
    *   **데이터 노드:** 실제 객체를 파일로 저장하고, 다중화 그룹을 이룸.

![데이터 저장소 컴포넌트](https://drive.google.com/thumbnail?id=1yZFv4JlJjX9QQ6TxpxKdaUXpBHCnVVNn&sz=w700)

*   **작은 객체 저장 문제 해결:** 작은 객체를 개별 파일로 저장하면 디스크 블록 낭비 및 inode 고갈 문제 발생. 이를 해결하기 위해 **WAL 방식**처럼 여러 작은 객체를 하나의 큰 읽기-쓰기 파일에 순차적으로 추가(append)하고, 꽉 차면 읽기 전용으로 전환.
    *   *객체 위치 찾기:* 파일 내 객체의 시작 오프셋과 크기를 기록한 매핑 테이블(RDB)을 각 데이터 노드마다 유지.

![작은 객체들을 한 파일에 저장하는 방안](https://drive.google.com/thumbnail?id=1l5kb_8A8rFW9T_OdnjmAET9tiB5Fz-Y7&sz=w600)
![개선된 데이터 저장 흐름](https://drive.google.com/thumbnail?id=1QrvUTzGSo2xUyOxMhJH0jHmkkxxr3NBm&sz=w800)

*   **데이터 내구성 보장:**
    *   **다중화(Replication):** 3중 복제 등으로 높은 내구성을 달성하지만 비용 오버헤드가 큼.
    *   **소거 코드(Erasure Code):** 데이터를 분할하고 패리티(parity) 블록을 생성해 분산 저장. 예: (8+4) 설정 시 12개 노드에 분산. 저장 비용이 낮고 내구성이 더 높지만, 계산(CPU) 비용과 읽기 지연(여러 노드에서 조합 필요)이 발생.

![(8+4) 소거 코드](https://drive.google.com/thumbnail?id=1eQ4rwbam6c-J75d1Ljo2dAqbCARJj7tB&sz=w800)

*   **데이터 정확성 (Data Integrity):**
    *   데이터 훼손 검증을 위해 **체크섬(Checksum)** 사용. 파일 생성 시 또는 주기적으로 체크섬을 비교하여 에러 감지 시 소거 코드나 다중화된 사본으로 복구.

*   **성능 및 부가 기능:**
    *   **멀티파트 업로드:** 대형 객체를 작게 쪼개어 병렬 업로드 후, 완료 시 조립 (ETag 검증 활용).
    *   **쓰레기 수집 (Garbage Collection):** 지연된 삭제, 실패한 멀티파트 업로드 조각 등을 정리하기 위해 유효한 객체만 새 파일로 복사(Compaction)하고 옛 파일을 삭제.

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)

*   **다중화 (Replication) vs 소거 코드 (Erasure Code)**
    *   **다중화:** 구현이 단순하고 읽기/쓰기 성능이 우수. 장애 복구가 빠름. 하지만 저장 용량 오버헤드(200%)가 커서 100PB 규모의 객체 저장소에서는 비용 문제가 발생.
    *   **소거 코드:** 오버헤드(50%)를 획기적으로 줄이면서도 내구성(11 Nines)을 극대화. 하지만 패리티 계산에 CPU 자원이 소모되고, 데이터를 조립하기 위해 여러 노드와 통신해야 하므로 읽기 지연이 증가. 객체 저장소(Cold Data 중심)의 특성을 고려하여 **소거 코드 도입은 비용 최적화를 위한 핵심 결정**.
*   **메타데이터 DB 스키마: 객체 ID vs 객체 이름(URI) 샤딩**
    *   *버킷 ID 샤딩:* 핫스팟 문제 발생 (유명 버킷에 트래픽 집중).
    *   *객체 ID 샤딩:* ID로 찾기는 좋으나, '주어진 버킷과 접두어에 해당하는 객체 목록(ls)' 같은 URI 기반 질의 처리가 매우 어려움.
    *   *해결책:* `bucket_id`와 `object_name`을 조합한 해시값을 샤딩 키로 사용. 읽기/쓰기 분산은 이루지만 범위 검색(ls)을 위해서는 여러 샤드에 질의를 브로드캐스트하여 취합(Scatter-Gather)하는 트레이드오프 감수.
*   **작은 파일 개별 저장 vs 큰 파일 병합 (Append-only)**
    *   개별 저장은 구현이 간단하나 블록(최소 4KB) 낭비와 inode 고갈을 야기.
    *   큰 파일 병합 방식은 공간 활용을 극대화하지만, 객체의 정확한 '오프셋'과 '크기'를 추적하기 위한 추가 매핑 구조(로컬 DB)가 필요하고, 이후 삭제된 객체로 인한 구멍(Fragmentation)을 메우기 위해 복잡한 가비지 컬렉션(Compaction) 과정이 필수적임.

## 3. 면접 예상 질문 (Interview Q&A)

**Q. 객체 저장소에서 아주 작은 파일(예: 수십 KB) 수백억 개를 저장할 때 발생할 수 있는 문제와 해결책은 무엇입니까?**
A. 일반적인 파일 시스템에 각 파일을 개별 저장하면, 파일 시스템의 최소 블록 크기(예: 4KB) 단위로 저장되어 공간이 낭비되고, 메타데이터를 저장하는 inode가 고갈되어 디스크 용량이 남아도 더 이상 파일을 저장할 수 없는 문제가 생깁니다. 이를 해결하기 위해 작은 객체들을 모아 하나의 큰 파일(Data File)에 순차적으로 추가(Append-only)하는 방식을 사용합니다. 이후 데이터 노드 내에 별도의 로컬 매핑 테이블(DB)을 두어 큰 파일 안에서 각 객체의 시작 오프셋과 크기를 기록해 빠르게 조회합니다.

**Q. 객체 저장소의 데이터 내구성을 99.9999% 이상으로 보장하기 위한 방법은 무엇입니까? Replication과 Erasure Code를 비교해 설명해주세요.**
A. 하드웨어 장애와 데이터센터(AZ) 장애를 대비해 데이터를 물리적으로 분리된 여러 장애 도메인에 분산 저장해야 합니다.
Replication(다중화)는 원본 데이터를 그대로 2~3곳에 복제합니다. 빠르고 구조가 단순하지만 200% 수준의 저장소 비용 오버헤드가 발생합니다.
Erasure Code(소거 코드)는 데이터를 N개의 조각으로 나누고 M개의 패리티 조각을 추가로 만들어 분산 저장합니다. 저장소 오버헤드는 50% 수준으로 크게 줄어들고 더 높은 내구성을 보장하지만, 읽고 쓸 때 CPU 계산 비용이 들고 장애 복구나 읽기 시 여러 노드에서 조각을 모아야 하는 지연(Latency) 트레이드오프가 있습니다. S3 같은 객체 저장소는 비용 효율성이 중요하므로 소거 코드를 적극 활용합니다.

**Q. S3에서 파일 일부가 조용히 손상되는 '비트 롯(Bit Rot)' 현상을 어떻게 감지하고 복구할 수 있나요?**
A. 데이터를 저장할 때 데이터의 해시값을 이용해 '체크섬(Checksum)'을 계산하여 함께 저장합니다. 파일을 읽을 때마다 읽은 데이터로 체크섬을 다시 계산하여, 저장된 체크섬과 비교합니다. 값이 다르면 데이터가 손상된 것으로 판단하고, 소거 코드의 다른 조각들이나 다중화된 다른 사본을 이용해 데이터를 원상태로 복구(Self-healing)합니다.

**Q. 멀티파트 업로드(Multipart Upload)란 무엇이며 어떤 이점이 있나요?**
A. 수 GB에 달하는 대용량 객체를 한 번에 업로드하면 중간에 네트워크 오류 발생 시 처음부터 다시 올려야 합니다. 멀티파트 업로드는 클라이언트가 큰 파일을 작은 조각(Part)으로 나누어 병렬로 업로드하는 방식입니다. 업로드 속도가 빨라지고, 실패한 조각만 재시도하면 되므로 안정성이 높아집니다. 업로드가 완료되면 서버 측에서 조각들의 ETag(MD5)를 검증하고 조립하여 하나의 객체로 만듭니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)

*   **메타데이터 DB의 분산 쿼리 최적화:** `bucket_name`과 `object_name`의 결합 해시로 샤딩했을 때, `s3 ls` 명령어처럼 특정 디렉터리 경로 하위의 모든 객체를 조회하는 쿼리의 성능은 어떻게 높일 수 있을까? (예: 별도의 검색 엔진(Elasticsearch 등) 연동, 캐싱 전략, 보조 인덱싱 등)
*   **쓰레기 수집(GC)과 시스템 부하:** 지연 삭제 및 공간 회수를 위해 수행되는 객체 복사(Compaction) 과정은 디스크 I/O와 네트워크 대역폭을 많이 소모한다. 실제 운영 환경에서 클라이언트의 정상적인 읽기/쓰기 성능에 영향을 주지 않으면서 GC를 백그라운드에서 안전하게 수행하기 위한 스로틀링(Throttling) 및 스케줄링 전략은 무엇이 있을까?
*   **강한 일관성(Strong Consistency) vs 최종 일관성(Eventual Consistency):** 예전 S3는 최종 일관성을 제공했지만 최근 강한 일관성을 지원하게 되었다. 분산 객체 저장소에서 새로 생성되거나 업데이트된 객체에 대해 즉각적인 읽기 일관성(Read-after-Write Consistency)을 보장하기 위해서는 메타데이터 저장소와 데이터 저장소 간의 트랜잭션 및 복제 지연을 어떻게 제어해야 할까?

## 5. 추가 학습 추천
*   **Erasure Code의 수학적 원리:** Reed-Solomon 코드가 패리티 블록을 어떻게 생성하고 복구하는지 구체적인 원리 파악.
*   **LSM-Tree 구조와 Compaction:** 작은 데이터를 모아 큰 파일로 만들고, 지워진 데이터를 정리하는 과정은 LevelDB나 RocksDB에서 쓰이는 LSM-Tree의 개념 및 Compaction 과정과 유사하므로 이 부분을 공부하면 이해도가 높아짐.
*   **Amazon S3 Architecture Deep Dive:** AWS re:Invent 영상이나 관련 백서를 통해 S3의 일관성 모델 변화 및 내부 스토리지 계층(Tiers)에 대한 연구.
*   **Ceph, MinIO:** 오픈소스 객체 저장소인 Ceph(RADOS)나 MinIO의 내부 아키텍처 다큐먼트 읽어보기.
