## 1. 저장소 시스템 101

### 💾 저장소 시스템의 3가지 유형

#### 1️⃣ 블록 저장소 (Block Storage)

**개념**
- HDD, SSD처럼 서버에 물리적으로 연결되는 형태
- **원시 블록(raw block)** 을 **볼륨(volume)** 형태로 제공

**특징**
- 서버가 원시 블록을 포맷하여 파일 시스템으로 사용
- 애플리케이션에 블록 제어권을 직접 부여 가능
- 네트워크를 통한 연결 지원
    - FC (Fibre Channel)
    - iSCSI

**장점**
- **최대 성능** - 직접 제어로 성능 극대화
- 데이터베이스, 가상 머신 엔진에 최적

---

#### 2️⃣ 파일 저장소 (File Storage)

**개념**
- 블록 저장소 위에 구현된 **추상화 계층**
- 파일과 디렉터리를 손쉽게 다루도록 지원

**특징**
- **계층적 디렉터리** 구조로 데이터 보관
- 네트워크 프로토콜 지원
    - SMB/CIFS
    - NFS
- 여러 서버에서 동시 접근 가능

**장점**
- **단순하고 사용하기 편리**
- 블록 제어, 볼륨 포맷 등 복잡한 작업 불필요

---

#### 3️⃣ 객체 저장소 (Object Storage)

**개념**
- **새로운 형태**의 저장소
- 데이터 영속성↑, 대규모 지원, 비용↓를 위해 **의도적으로 성능 희생**

**특징**
- 모든 데이터를 **수평적 구조**로 객체 보관
- **RESTful API**를 통한 접근
- **'차가운(cold)' 데이터** 보관에 초점
- 다른 저장소에 비해 **상대적으로 느림**

**사용 사례**
- 데이터 아카이브
- 백업
- 실시간 갱신 불필요한 데이터

**대표 서비스**: AWS S3

---

### 📌 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| **버킷 (Bucket)** | 객체를 보관하는 논리적 컨테이너<br>전역적으로 유일해야 함 |
| **객체 (Object)** | 버킷에 저장하는 개별 데이터<br>데이터(페이로드) + 메타데이터 |
| **버전 (Versioning)** | 한 객체의 여러 버전을 같은 버킷에 보관<br>실수로 삭제/덮어쓴 객체 복구 가능 |
| **URI** | 각 객체를 고유하게 식별하는 RESTful API 엔드포인트 |
| **SLA** | Service-Level Agreement<br>서비스 제공자-클라이언트 간 계약 |

## 2. 1단계: 설계 범위 확정

### ✅ 기능 요구사항

```
1️⃣ 버킷 생성
2️⃣ 객체 업로드 및 다운로드  
3️⃣ 객체 버전 관리
4️⃣ 버킷 내 객체 목록 출력
```

### ✅ 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| **저장 용량** | 100PB |
| **데이터 내구성** | 99.9999% (식스 나인) |
| **서비스 가용성** | 99.99% (포 나인) |
| **저장소 효율성** | 높은 안정성 + 성능 보증<br>저장소 비용 최소화 |

### 📊 개략적인 규모 추정

#### 디스크 용량

**객체 크기 분포**
```
📦 소형 (20%): 1MB 미만
📦 중형 (60%): 1MB ~ 64MB  
📦 대형 (20%): 64MB 이상
```

**계산 기준**
- 소형 객체: 0.5MB
- 중형 객체: 32MB
- 대형 객체: 200MB

**결과**
```
총 객체 수: 6억 8천만 개
총 용량: 100PB
메타데이터: 0.68TB (객체당 1KB 가정)
```

#### IOPS

**가정**
- SATA 인터페이스, 7200rpm HDD
- 초당 100~150회 임의 데이터 탐색 가능
- 100~150 IOPS

## 3. 2단계: 개략적 설계안

### 🔑 핵심 특징

#### 1. 객체 불변성 (Object Immutability)

> **객체 저장소와 다른 저장소의 가장 큰 차이**

```
❌ 점진적 변경 불가능
✅ 삭제 후 새 버전으로 완전 대체만 가능
```

#### 2. 키-값 저장소 (Key-Value Store)

```
URI (키) → 데이터 (값)
s3://mybucket/script.txt → 객체 데이터
```

#### 3. 읽기 패턴

```
저장: 1회
읽기: 여러 번
```

#### 4. 다양한 객체 크기 지원

소형부터 대형까지 모든 크기 처리

### 🏗️ 시스템 아키텍처

```
                    ┌─────────────┐
                    │   Client    │
                    └──────┬──────┘
                           │
                           ▼
                  ┌─────────────────┐
                  │ Load Balancer   │
                  └────────┬────────┘
                           │
                           ▼
                  ┌─────────────────┐
                  │   API Service   │
                  └─┬─────┬─────┬───┘
                    │     │     │
        ┌───────────┘     │     └───────────┐
        ▼                 ▼                 ▼
   ┌────────┐      ┌────────────┐    ┌──────────┐
   │  IAM   │      │  Metadata  │    │   Data   │
   │Service │      │  Storage   │    │ Storage  │
   └────────┘      └────────────┘    └──────────┘
```

### 📦 주요 컴포넌트

#### Load Balancer
```
역할: RESTful API 요청을 API 서버들에 분산
```

#### API Service
```
역할: IAM, 메타데이터, 저장소 서비스 호출 조율
```

#### IAM Service
```
기능:
├─ 인증 (Authentication): 호출 주체 확인
├─ 권한 부여 (Authorization): 작업 수행 가능 여부 검증  
└─ 접근 제어 (Access Control): 중앙 관리
```

#### Data Storage
```
역할: 실제 객체 데이터 보관 및 제공
```

#### Metadata Storage
```
역할: 객체 메타데이터 보관
```

> **💡 참고**: 데이터 저장소와 메타데이터 저장소는 **논리적 구분**이며,  
> 구현 방법은 다양할 수 있습니다.

---

### 📤 객체 업로드 과정

```
1. 클라이언트
   └─> HTTP PUT /bucket-to-share (버킷 생성)

2. API Service
   └─> IAM 호출: WRITE 권한 확인

3. API Service  
   └─> 메타데이터 저장소: 버킷 정보 등록

4. 클라이언트
   └─> HTTP PUT /bucket-to-share/script.txt (객체 생성)

5. API Service
   └─> IAM 호출: 신원 및 WRITE 권한 확인

6. API Service
   └─> 데이터 저장소: 객체 데이터 전송

7. 데이터 저장소
   └─> 객체 저장 후 UUID 반환

8. API Service
   └─> 메타데이터 저장소: 새 항목 등록
       (object_id, bucket_id, object_name 등)
```

---

### 📥 객체 다운로드 과정

**특징**
- 버킷은 계층 구조 미지원
- 하지만 **논리적 계층** 구현 가능
    - 예: `버킷이름/폴더/파일`

**호출 사례**

```
1. 클라이언트
   └─> GET /bucket-to-share/script.txt

2. Load Balancer
   └─> API 서버로 전달

3. API Service
   └─> IAM: READ 권한 확인

4. API Service  
   └─> 메타데이터 저장소: 객체 UUID 조회

5. API Service
   └─> 데이터 저장소: UUID로 객체 데이터 조회

6. API Service
   └─> HTTP GET 응답으로 객체 데이터 반환
```

## 4. 3단계: 상세 설계

### 🗄️ 데이터 저장소 아키텍처

```
        API Service
             │
             ▼
    ┌────────────────────┐
    │ Data Routing       │ ◄─ 무상태 RESTful/gRPC
    │ Service            │    수평 확장 가능
    └─────────┬──────────┘
              │
              ▼
    ┌────────────────────┐
    │ Placement          │ ◄─ 최적 노드 선택
    │ Service            │    가상 클러스터 지도
    └─────────┬──────────┘    Paxos/Raft 합의
              │
              ▼
    ┌────────────────────┐
    │ Data Nodes         │
    │ ┌────────────────┐ │
    │ │Primary (주)    │ │ ◄─ 실제 데이터 저장
    │ └────────────────┘ │    다중화 그룹
    │ ┌────────────────┐ │    Heartbeat 전송
    │ │Secondary (부)  │ │
    │ └────────────────┘ │
    │ ┌────────────────┐ │
    │ │Secondary (부)  │ │
    │ └────────────────┘ │
    └────────────────────┘
```

---

### 🚀 데이터 라우팅 서비스

**특징**
- RESTful 또는 gRPC 서비스 제공
- **무상태 서비스** - 서버 추가로 쉽게 확장

**주요 기능**
```
1. 배치 서비스를 호출하여 최적 데이터 노드 판단
2. 데이터 노드에서 데이터 읽어 API 서비스에 반환
3. 데이터 노드에 데이터 기록
```

---

### 🎯 배치 서비스

**역할**
- 어느 데이터 노드에 데이터를 저장할지 결정

**가상 클러스터 지도**
```
내부 유지 정보:
├─ 클러스터의 물리적 형상 정보
├─ 데이터 노드 위치 정보
└─ 데이터 사본의 물리적 분리 보장
```

> **💡 핵심**: 물리적 분리는 **높은 데이터 내구성**의 핵심 요소

**고가용성 보장**
```
구성: 5~7개 노드 클러스터
프로토콜: Paxos 또는 Raft 합의 프로토콜

장점:
└─ 일부 노드 장애 시에도 서비스 지속
   (건강한 노드 > 클러스터 크기의 절반)
```

---

### 💾 데이터 노드

**역할**
- 실제 객체 데이터 보관

**다중화 그룹**
```
구성: 1개 주 노드 + 2개 부 노드
목적: 데이터 안정성과 내구성 보증
```

**Heartbeat 메시지**

주기적으로 배치 서비스에 전송하는 정보:
```
1. 부착된 디스크 드라이브(HDD/SSD) 수
2. 각 드라이브에 저장된 데이터 양
3. 데이터 사본 보관 위치
```

│ ┌──────────────────────────┐ │
│ │ 로컬 저장                 │ │

**조회 연산 구현**
- UUID 입력 → 다중화 그룹 반환

**⚖️ Trade-off**

```
강력한 데이터 일관성 보장
  ↕
높은 응답 지연

└─ 가장 느린 사본 완료까지 대기
```

### 📁 데이터 저장 방식

#### ❌ 문제: 객체를 개별 파일로 저장

**작은 파일이 많을 때 발생하는 문제**

**1. 데이터 블록 낭비**
```
파일 시스템: 파일을 별도 디스크 블록에 저장
문제: 작은 파일도 블록 하나를 온전히 사용
결과: 낭비되는 블록 증가
```

**2. 아이노드(inode) 고갈**
```
아이노드: 파일 위치 정보를 저장하는 특별한 블록
문제: 사용 가능한 수가 디스크 초기화 시 결정
      작은 파일 수백만 개 → 아이노드 소진 가능
```

---

#### ✅ 해결: WAL(Write-Ahead Log) 방식

**개념**
- 객체를 이미 존재하는 파일에 추가
- 작은 객체들을 큰 파일 하나로 모음

**동작 방식**

```
┌──────────────────────────────────┐
│ Read-Write File (현재 쓰기 중)   │
├──────────────────────────────────┤
│ Obj1 | Obj2 | Obj3 | Obj4 | ...  │
└──────────────────────────────────┘
         │
         │ 용량 임계치 도달
         ▼
┌──────────────────────────────────┐
│ Read-Only File (읽기 전용)       │
├──────────────────────────────────┤
│ Obj1 | Obj2 | Obj3 | Obj4 | ...  │
└──────────────────────────────────┘
         +
┌──────────────────────────────────┐
│ Read-Write File (새 파일 생성)   │
├──────────────────────────────────┤
│ Obj5 | Obj6 | ...               │
└──────────────────────────────────┘
```

**주의사항**
```
쓰기 연산: 순차적으로 수행 필요
문제: 여러 CPU 코어 병렬 쓰기 시 내용 뒤섞임 가능
```

**최적화: CPU 코어별 전담 파일**

```
CPU Core 1 → Read-Write File 1
CPU Core 2 → Read-Write File 2
CPU Core 3 → Read-Write File 3
CPU Core 4 → Read-Write File 4

장점: 쓰기 대역폭 극대화
```

### 🔍 객체 소재 확인

**문제**
```
데이터 파일 안에 많은 작은 객체 존재
→ UUID로 어떻게 위치를 찾을까?
```

**필요 정보**
```
1. 객체가 보관된 데이터 파일
2. 데이터 파일 내 객체 오프셋
3. 객체 크기
```

**object_mapping 테이블**

| 필드 | 설명 |
|------|------|
| `object_id` | 객체의 UUID |
| `file_name` | 객체를 보관하는 파일 이름 |
| `start_offset` | 파일 내 객체 시작 주소 |
| `object_size` | 객체 크기 (바이트) |

|------|----------|----------|----------|
| **RocksDB** | SSTable 기반 | ⭐⭐⭐ 우수 | ⭐ 느림 |
| **RDBMS** | B+ 트리 기반 | ⭐ 느림 | ⭐⭐⭐ 우수 |

**✅ 선택: 관계형 데이터베이스 (RDBMS)**

**이유**
```
1. 한 번 기록 후 변경 없음
2. 읽기 연산이 매우 빈번
3. 데이터 노드마다 독립적 설치 가능
   (다른 노드와 공유 불필요)
```

---

### 🔄 개선된 데이터 저장 흐름

```
1. API Service
   └─> 새 객체 저장 요청

2. Data Node Service
   ├─> Read-Write 파일 마지막에 추가
   └─> object_mapping 테이블에 레코드 추가
       (object_id, file_name, start_offset, object_size)

3. Data Node Service  
   └─> UUID 반환
```

## 5. 데이터 내구성

> **목표**: 99.9999% 수준의 데이터 내구성 달성

### 🎯 달성 방법

```
장애 발생 가능한 모든 경우 고려
    +
데이터 적절히 다중화
    ↓
높은 내구성 보장
```

---

### 1️⃣ 하드웨어 장애와 장애 도메인

**문제**
```
드라이브 한 대로는 내구성 목표 달성 불가능
```

**해결**
```
데이터를 여러 드라이브에 복제
→ 특정 드라이브 장애가 전체 가용성에 영향 없도록
```

**3중 복제 시**
```
내구성: 약 0.999999 (99.9999%)
```

**장애 도메인 (Failure Domain)**
```
정의: 중요한 서비스 문제 발생 시
      부정적 영향을 받는 물리적/논리적 구획

고려사항: 여러 장애 도메인 영향을 복합적으로 평가
```

---

### 2️⃣ 다중화 (Replication)

**개념**
```
원본 데이터 ──┬──> 복제본 1 (다른 서버)
              ├──> 복제본 2 (다른 서버)
              └──> 복제본 3 (다른 서버)
```

**특징**
```
내구성: 99.9999% (3중 복제)
저장 오버헤드: 200%
계산 자원: 불필요
```

**장점**
- 빠른 읽기 (1개 노드만 접근)
- 복제만 하면 되므로 쓰기도 빠름

**단점**
- 높은 저장 비용 (200% 오버헤드)

---

### 3️⃣ 소거 코드 (Erasure Coding)

**개념**
```
데이터를 작은 단위로 분할
    +
패리티(parity) 정보 생성
    ↓
일부 소실 시 복구 가능
```

**동작 원리 - (4+2) 예시**

```
Step 1: 데이터 분할
원본 ──> [d1][d2][d3][d4]

Step 2: 패리티 생성  
수학 공식 적용
    ↓
[p1][p2]

Step 3: 분산 저장
서버1: d1    서버4: d4
서버2: d2    서버5: p1
서버3: d3    서버6: p2

Step 4: 복구 (d3, d4 소실 시)
d1, d2, p1, p2 + 수식
    ↓
d3, d4 복원 가능!
```

**(8+4) 소거 코드**
```
8개 데이터 블록 + 4개 패리티 블록
내구성: 99.9999999999%
저장 오버헤드: 50%
```

**장점**
- 낮은 저장 비용 (50% 오버헤드)
- 매우 높은 내구성

**단점**
- 느린 읽기 (최대 8개 노드 접근 필요)
- 패리티 계산으로 쓰기 성능 저하

---

### ⚖️ 다중화 vs 소거 코드 비교

|  | 다중화 | 소거 코드 |
|--|--------|-----------|
| **내구성** | 99.9999%<br>(3중 복제) | 99.9999999999%<br>(8+4 코드) |
| **저장 오버헤드** | 200% | 50% |
| **계산 자원** | 불필요 | 패리티 계산 필요 |
| **쓰기 성능** | ⭐⭐⭐<br>복제만 수행 | ⭐<br>패리티 계산으로 지연 |
| **읽기 성능** | ⭐⭐⭐<br>1개 노드만 접근 | ⭐<br>다중 노드 접근 필요 |
| **적합한 경우** | 응답 지연이 중요 | 저장 비용이 중요 |

---

### 4️⃣ 정확성 검증 (Checksum)

**문제**
```
대규모 시스템에서는:
├─ 디스크 데이터 훼손
└─ 메모리 데이터 훼손 ← 자주 발생!
```

**해결: 체크섬 (Checksum)**
```
정의: 데이터 에러 발견용 작은 크기 블록
위치: 프로세스 경계
```

**동작 방식**

```
송신측
  ├─ 데이터: [원본 데이터]
  └─ 체크섬 계산: [원본 체크섬]

전송
  ↓

수신측  
  ├─ 데이터 수신: [수신 데이터]
  ├─ 체크섬 재계산: [새 체크섬]
  └─ 비교
      ├─ 일치 → ✅ 데이터 정상
      └─ 불일치 → ❌ 데이터 훼손
```

**체크섬 알고리즘**
```
MD5, SHA1, HMAC 등
```

---

### 🔐 소거 코드 + 체크섬 결합

**(8+4) 소거 코드 + 체크섬 활용 절차**

```
1. 객체 데이터 + 체크섬 가져오기
   
2. 각 조각별 처리
   ├─ 수신 데이터 체크섬 계산
   ├─ 원본 체크섬과 비교
   │   ├─ 일치 → 정상
   │   └─ 불일치 → 다른 장애 도메인에서 재시도
   └─ 8조각 모두 수신할 때까지 반복

3. 원본 객체 복원
   
4. 클라이언트에게 전달
```

**장점**
```
소거 코드: 일부 소실 시 복구 가능
체크섬: 데이터 정확성 검증
    ↓
높은 내구성 + 데이터 무결성 보장
```

저장은 OK, 하지만...

### 📊 object 테이블 규모 확장

**특징**

```

**샤딩 키 선택**

#### ❌ Option 1: bucket_id로 샤딩
```
문제: 핫스팟 샤드 발생
예시: 한 버킷에 수십억 개 객체
     → 특정 샤드에 부하 집중

#### ❌ Option 2: object_id로 샤딩
```
장점: 부하 균등 분산

```
선택 이유:
└─ 대부분의 메타데이터 연산이
   객체 URI 기준으로 수행되기 때문

예시: s3://mybucket/path/to/file.txt
      bucket_name: mybucket
      object_name: path/to/file.txt
      샤딩 키: hash(mybucket + path/to/file.txt)
```

**장점**
```
✅ 부하 균등 분산
✅ 효율적인 조회
```

### 📂 버킷 내 객체 목록 확인

```
디렉터리처럼 데이터 정리 가능

```

## 7. 성능 최적화 및 운영

### 🔄 객체 버전 관리

**기능**
```
한 객체의 여러 버전을 같은 버킷에 보관
→ 실수로 삭제/덮어쓴 객체 복구 가능
```

**삭제 메커니즘**

```
타임라인:
v1: script.txt (최초 업로드)
v2: script.txt (수정)
v3: script.txt (또 수정)
v4: [DELETE MARKER] ← 삭제 표시 추가

모든 버전이 버킷에 보관됨!
```

**삭제 표식 (Delete Marker)**
```
특징:
├─ 객체의 새로운 버전으로 취급
├─ 삽입 순간 '현재 버전'이 됨
└─ 실제 데이터는 삭제 안 됨

동작:
GET script.txt 
  → 404 Object Not Found
  (현재 버전이 삭제 표식이므로)

GET script.txt?versionId=v2
  → 성공 (이전 버전 접근 가능)
```

### 🚀 멀티파트 업로드 (Multipart Upload)

**문제**
```
대용량 객체 (수 GB 이상) 업로드:
├─ 시간이 오래 걸림
└─ 네트워크 문제 시 처음부터 재시작
```

**해결**
```
객체를 작은 조각으로 분할
    ↓
독립적으로 병렬 업로드
    ↓
모든 조각 완료 후 원본 복원
```

**동작 방식**

```
대용량 파일 (예: 10GB)
    │
    ├─> Part 1 (100MB) ─┐
    ├─> Part 2 (100MB) ─┤
    ├─> Part 3 (100MB) ─┼─> 병렬 업로드
    ├─> Part 4 (100MB) ─┤
    └─> ...            ─┘
            │
            ▼
    모든 조각 완료 확인
            │
            ▼
    객체 저장소가 자동 병합
            │
            ▼
    원본 객체 복원 완료
```

**장점**
```
✅ 업로드 시간 단축 (병렬 처리)
✅ 실패한 조각만 재전송 (전체 재업로드 불필요)
✅ 네트워크 장애에 강건
```

### 🗑️ 쓰레기 수집 (Garbage Collection)

**정의**
```
더 이상 사용되지 않는 데이터의
저장 공간을 자동으로 회수하는 절차
```

**수집 대상 (쓰레기 데이터)**

```
1️⃣ 객체의 지연된 삭제
   └─ 삭제 표시만 되고 실제로 안 지워진 데이터

2️⃣ 갈 곳 없는 데이터
   ├─ 반쯤 업로드된 데이터
   └─ 취소된 멀티파트 업로드 데이터

3️⃣ 훼손된 데이터
   └─ 체크섬 검사 실패 데이터
```

**동작 방식**

```
즉시 삭제 ❌
    ↓
주기적 정리 메커니즘 실행 ✅
    ↓
쓰레기 데이터 안전하게 제거
```

**복제본 처리**

```
다중화 사용 시:
└─ 주 저장소 노드
   부 저장소 노드 1
   부 저장소 노드 2
   모두에서 삭제!

소거 코드 (8+4) 사용 시:
└─ 12개 노드 모두에서 삭제!
```

### 📋 버킷 내 객체 목록 출력 최적화

**기본 접근 방식**
```

    ↓
결과 취합
    ↓
정렬 및 페이징

**페이지네이션 복잡성**

문제:
├─ 샤드별로 반환 객체 수 다름
├─ 샤드별 오프셋 추적 필요

Cursor: {s1:10, s2:15, s3:8}
```

**최적화 전략**

```
Option 1: 목록 성능 우선순위 낮춤
└─ 객체 저장소는 규모/내구성 최적화에 집중

Option 2: 비정규화
└─ 버킷 ID로 샤딩하는 별도 테이블 생성
   목록 전용 데이터 저장
```

## 8. 4단계: 마무리

### 🎯 핵심 요약

```

#### 비기능 요구사항 달성
```
✅ 100PB 데이터 저장
✅ 99.9999% 데이터 내구성
✅ 99.99% 서비스 가용성
✅ 저장소 효율성 최적화
```

---

### 📚 주요 설계 결정사항

#### 1. 데이터 저장소

**아키텍처**
```
Data Routing Service (무상태, 확장 가능)
    ↓
Placement Service (Paxos/Raft 합의)
    ↓
Data Nodes (다중화 그룹)
```

**데이터 영속성**
```
WAL 방식 채택
├─ CPU 코어별 전담 Read-Write 파일
├─ 임계치 도달 시 Read-Only 전환
└─ object_mapping 테이블로 위치 추적
```

---

#### 2. 데이터 내구성

**두 가지 방안**

| 방안 | 선택 기준 |
|------|-----------|
| **다중화** | 응답 지연이 중요한 경우 |
| **소거 코드** | 저장 비용이 중요한 경우 |

**정확성 보장**
```
체크섬 검증 메커니즘
└─ 프로세스 경계마다 데이터 무결성 확인
```

---

#### 3. 메타데이터 저장소

**스키마 설계**
```
bucket 테이블: DB 복제본으로 읽기 분산
object 테이블: bucket_name + object_name으로 샤딩
```

**샤딩 전략**
```
장점:
├─ 부하 균등 분산
├─ 효율적인 객체 조회
└─ URI 기반 연산에 최적화
```

**목록 출력**
```
복잡성 인정
└─ 성능 보장보다 규모/내구성 우선
```

---

#### 4. 성능 최적화

**멀티파트 업로드**
```
대용량 파일을 조각으로 분할
    ↓
병렬 업로드로 시간 단축
    ↓
실패 시 조각만 재전송
```

**쓰레기 수집**
```
주기적 실행
└─ 사용되지 않는 저장 공간 회수
```

---

### 💡 핵심 설계 원칙

#### 1. Trade-off 이해

```
일관성 ↔ 지연시간
├─ 강력한 일관성: 모든 복제 완료 후 응답
└─ 낮은 지연: 일부만 완료 후 응답

내구성 ↔ 비용
├─ 다중화: 높은 비용, 빠른 성능
└─ 소거 코드: 낮은 비용, 느린 성능

읽기 성능 ↔ 저장 효율
├─ 다중화: 빠른 읽기, 200% 오버헤드
└─ 소거 코드: 느린 읽기, 50% 오버헤드
```

---

#### 2. 장애 대응

```
다중화 그룹
└─ 물리적으로 분리된 위치에 복제

체크섬 검증
└─ 데이터 훼손 조기 발견

합의 프로토콜
└─ 배치 서비스 고가용성

장애 도메인 고려
└─ 복합적 장애 시나리오 대비
```

---

#### 3. 규모 확장성

```
무상태 서비스 설계
└─ Data Routing Service
   API Service

적절한 샤딩 전략
└─ bucket_name + object_name

메타데이터 분리
└─ 데이터와 메타데이터 독립적 확장
```

---

#### 4. 운영 효율

```
주기적 쓰레기 수집
└─ 자동 공간 회수

멀티파트 업로드
└─ 대용량 파일 효율적 처리

버전 관리
└─ 데이터 복구 용이

Heartbeat 모니터링
└─ 노드 상태 실시간 파악
```
