## 호텔 예약 시스템

- 이번 장에서는 메리어트 인터내셔널 같은 호텔 체인의 예약 시스템을 설계할 것이다.

---

### 1. 문제 이해 및 설계 범위 확정

- 호텔 예약 시스템은 복잡하고 그 컴포넌트는 시스템을 사업에 어떻게 이용할지에 따라 달라진다.

#### 요구사항 정리

- 5000개 호텔에 100만개 객실을 갖춘 호텔 체인을 위한 웹사이트 구축한다고 가정
- 대금은 예약때 전부 지불한다.
- 웹, 앱에서만 가능
- 예약 취소 가능
- 10% 초과 예약 가능(예약취소 할 것을 고려하여)

#### 계략적 규모 추정

- 일일 예상 예약 건수 240,000
- 초당 예약 트랜잭션 수(TPS)는 그다지 높지 않다.

1. 호텔/객실 상세 페이지 사용자 조회
2. 예약 상세 정보 페이지(날짜, 투숙인원, 결제 방법등) 조회
3. 객실 예약 페이지 사용자가 `예약` 버튼을 눌러 객실을 예약한다(트랜잭션 발생)

- 대략 10%의 사용자가 다음 단계로 진행하고 90%의 사용자는 최종 단계에 도달하기 전에 흐름을 이탈한다고 하자
    - 객실 상세 페이지(QPS =300)
    - 예약 상세 페이지(QPS =30)
    - 객실 예약 페이지(QPS =3)
    - 최종 예약 TPS는 3이라는 것을 알고 있으므로, 그 수치에서 역산한 결과다.

--- 

### 2. 개략적 설계안 제시 및 동의 구하기

- API 설계
- 데이터 모델
- 개략적 설계안

#### API 설계

- API 설계중 researvationID는 이중 예약을 방지하고 동일한 예약은 단 한번만 이루어지도록 보증하는 멱등 키다.
    - (애플리케이션 단에서 미리 이중 예약을 방지하기 위한 방패)

#### 데이터 모델

- 관계형 데이터베이스는 읽기 빈도가 쓰기 연산에 비해 높은 작업 흐름을 잘 지원한다.
- 호텔 웹/앱을 방문하는 사용자의 수는 실제로 객실을 예약하는 사용자에 비해 압도적으로 많다.
- NoSQL 데이터베이스는 대체로 쓰기 연산에 최적화되어 있다.
- 관계형 데이터베이스는 읽기가 압도적인 작업 흐름은 충분히 잘 지원한다.
- ACID 속성은 예약 시스템을 만드는 경우 중요하다.
    - 이 속성이 만족되지 않으면 잔액이 마이너스가 되는 문제
    - 이중 예약 문제 등 방지 하기 어렵다.
- 관계형 데이터베이스를 사용하면 데이터를 쉽게 모델링할 수 있다. 비즈니스 데이터의 구조를 명확하게 표현할 수 이ㅆ다.
- 엔티티 간의 관계를 안정적으로 지원할 수 있다.

#### 상태값

- 결제 대기, 결제 완료, 환불 완료, 취소, 승인 실패의 다섯 상태 가운데 하나를 값을 가진다.
- 해당 테이블 설계에서 room_id가 있어 에어비앤비 같은 회사에는 적합하다 하지만 호텔의 경우에는 다르다.
- 사용자는 특정 객실을 예약하는 것이 아니라 특정 호텔의 특정 객실 유형을 예약한다.
    - 스탠다드 룸, 킹 사이즈 룸, 등이 될수 있다.
    - 객실 번호는 예약할 때가 아닌 투숙객이 체크인 하는 시점에 부여된다.

#### 개략적 설계안

- 이 호텔 예약 시스템에는 마이크로서비스 아키텍처를 사용한다.
- 공개 api 게이트웨이가 있고 각 서비스별로 나눠며 RDB 각 서비스별로 나뉜다.

#### 상세 설계

- 개선된 데이터 모델
- 동시성 문제
- 시스템의 규모 확장
- 마이크로서비스 아키텍처에서의 데이터 일관성 문제에 대한 해결 방안

#### room_type_inventory 테이블 설계

- 역할: 특정 호텔의 특정 객실 유형(room_type) 이 날짜별로 몇 개까지 예약 가능한지, 실제로 몇 개 예약됐는지 관리하는 핵심 테이블.
    - 컬럼 의미
        - hotel_id : 호텔 식별자
        - room_type_id : 객실 유형 식별자 (싱글, 더블, 스위트…)
        - date : 해당 일자
        - total_inventory : 그 날짜에 예약 가능한 객실 수(유지보수 등으로 제외된 객실 수는 뺀 값)
        - total_reserved : 그 날짜에 이미 예약된 객실 수
          PK(기본 키): (hotel_id, room_type_id, date) 복합 키
          ⇒ “어느 호텔의, 어떤 객실 유형이, 어떤 날짜에 얼마나 남았는지”를 한 레코드로 표현.

- 미리 데이터 채우기
  2년 치 모든 날짜에 대해, 호텔 × 객실유형 × 날짜 조합을 미리 만들어 두는 구조.

    - 예: 호텔 5,000개, 객실 유형 20종, 2년(365일 × 2)
    - → 5,000 × 20 × 2 × 365 ≒ 7,300만 행
    - → 하나의 DB에 넣기엔 크게 무리 없는 수준이지만,
    - 단일 DB만 쓰면 SPOF(단일 장애점)라서 가용영역/지역 단위로 복제해 두는 게 좋다는 설명.

#### 동시성 문제

- 이중 예약을 어떻게 방지할 것 인가 두가지 문제를 해결해야한다.
    1. 같은 사용자가 예약 버튼을 여러번 누를 수 있다.
    2. 여러 사용자가 같은 객실을 동시에 예약하려 할 수 있다.

1. 클라이언트 측에서 예약 버튼 클릭시 회색으로 표시 또는 숨기기, 비활성화 할수 있다.   
   하지만 사용자가 자바스크립트를 비활성화하면 클라이언트 측에서 절차는 우회 할수 있다.
2. 멱등성 키 추가하는 방법 몇번을 호출해도 같은 결과를 보장하기위해 reservation_id를 멱등 키로 사용하여  
   이중 예약 문제를 해결하는 방안이다.

#### 또 다른 방안

- 비관적 락
- 낙관적 락
- 데이터베이스 제약 조건

#### 비관적 락

- 비관적 동시성 제어 방안이라고도 부르며 사용자가 레코드를 갱신하려고 하는 순간 락을 걸어 동시 업데이트를 방지하는 기술이다.
- 해당 레코드를 갱신하려는 다른 사용자는 먼저 락을 건 사용자가 변경을 마치고 락을 해제할 때까지 기다려야 한다.

장점

- 애플리케이션이 변경중이거나 변경이 끝난 데이터를 갱신하는 일을 막을 수 있다.
- 구현이 쉽고 모든 갱신 연산을 직렬화하여 충돌을 막는다. 데이터에 대한 경합이 심할 때 유용하다.

단점

- 교착 상태가 발생할수 있다,
- 확장성이 낮다. 트랜잭션이 너무 오랫동안 락을 해제하지 않고 있으면 다른 트랜잭션은 락이 걸린 자원에 접근할수 없다.
- 많은 엔티티에 관련된 경우 데이터베이스 성능이 심각하게 영향을 끼친다.

이런 이유로 예약 시스템에 비관적 락 매커니즘은 사용하는 것은 권장하지 않는다.

#### 낙관적 락

- 낙관적 동시성 제어라고도 불리는 방안으로 여러 사용자가 동시에 같은 자원을 갱신하려 시도하는 것을 허용한다.
- 버전와 타임스탬프의 두가지 방법으로 구현한다. 서버 시계는 시간이 지남에 따라 부정확해질 수 있으므로 일반적으로 버전 번호를 선택한다.
- 낙관적 락은 일반적으로 비관적 락보다 빠르다. 데이터베이스에 락을 걸지 않기 때문이다.
- 하지만 동시성 수준이 아주 높으면 성능이 급격하게 나빠진다.
- 잔여 객실 수를 읽을 수 있는 클라이언트 수에 제한이 없으므로 모든 클라이언트는 같은 잔여 객실 수와 같은 버전 번호를 취득한다.
- 하지만 실제로 버전 번호 갱신에 성공하는 클라이언트는 오직 하나이다.
- 실패한 고객은 재시도를 통해 다시 예약한다. 최종 결과는 정확하겠지만 반복되는 재시도 때문에 사용자가 아주 불쾌한 경험을 하게 될것이다.

장점

- DB 자우너에 락을 걸 필요가 없다. 사실 데이터베이스 관점에서 보면 락이 없다. 버전 번호를 통해 데이터 일관성을 유지한다.
- 낙관적 락은 데이터에 대한 경쟁이 치열하지 않은 상황에 적합하다. 그런 상황에서는 락을 관리하는 비용 없이 트랜잭션을 실행할 수 있다.

단점

- 데이터에 대한 경쟁이 치열한 상황에서는 성능이 좋지 못하다.

낙관적락은 호텔 예약 시스템에 적합한 선택지다. 예약 QPS가 일반적으로는 높지 않기 때문이다.

#### 데이터베이스 제약 조건

CHECK (total_inventory - total_reserved >= 0) 제약을 걸어 두고, 예약 시 total_reserved가 재고(total_inventory)를  
넘어가면 제약 조건이 깨져서 트랜잭션이 오류로 실패하고 전체가 롤백되도록 만드는 방식이다.  
장점

- 구현이 간단하고 데이터에 대한 경쟁이 심하지 않을 때 잘 동작한다.
  단점
- 경쟁이 심하면 실패하는 연산 수가 엄청나게 늘어날 수 있다.
- 트랜잭션 실패 횟수가 크게 늘어나서 화면에는 예약 가능이지만 실제로는 객실이 없다가 떠서 사용자 경험이 나쁘다.
  구조는 단순하고 호텔 예약처럼 QPS가 낮고 경쟁이 크지 않은 시스템에서는 현실적으로 쓸만한 선택지

#### 샤딩

- 데이터베이스 규모를 늘리는 방법
- 이 시스템의 대부분의 질의는 hotel_id를 필터링 조건으로 사용한다. 따라서 해당 컬럼이 샤딩 조건으로 쓰면 좋다.
- 16개 샤드로 분산한다면 QPS가 30,000이면 샤딩 후에 각 샤드는 1875QPS를 처리하게 된다. 한대 Mysql 서버로 감당할 수 있는 부하다.

#### 캐시

- 호텔 잔여 객실 데이터는 과거가 중요하지 않고, 현재·미래만 중요하다.

- 그래서 오래된 데이터는 자동으로 사라지도록 TTL(Time-To-Live) 을 두고,
- 잔여 객실 조회는 가능하면 다른 DB 대신 캐시(예: Redis) 를 먼저 보도록 만드는 게 좋다.

- Redis는 TTL + LRU 정책을 지원해서, 메모리를 효율적으로 쓰면서
- 잔여 객실 조회·예약 로직 앞단의 캐시 계층으로 쓰기에 적합하다.

- 예약 규모가 커지면(booking.com / expedia 수준) 데이터베이스 앞에 캐시 계층을 둬서
- “잔여 객실 확인 + 예약 처리” 중 일부를 캐시가 처리하게 만들어 DB 부하를 줄일 수 있다.

- 다만, Redis 캐시만 보고 “잔여 객실 있음”이라고 보여도,
- 최종적으로는 데이터베이스에서 한 번 더 확인해야 한다.
- 잔여 객실 수에 대한 궁극적인 진실은 DB에 있기 때문.

#### 모놀리스 아키텍처

- 전통적인 모놀리스에선 보통 하나의 공유 관계형 DB를 써서 데이터 일관성을 확보한다.
- 이 책의 설계도 마이크로서비스를 도입했지만, 예약 서비스가 예약·잔여 객실 API를 모두 담당하고, 두 테이블을 같은 RDB에 저장하는 하이브리드 방식을 택했다.
- → RDB의 ACID 특성을 그대로 쓰면서 예약 처리 중 동시성 문제를 쉽게 해결하려는 선택.
- 다만 “진짜 마이크로서비스답게 하려면 서비스마다 독자적인 데이터베이스를 가져야 한다”는 관점에선,
- 이런 하이브리드 방식이 마이크로서비스의 순수성을 일부 포기한 타협안이라고 볼 수도 있다.

### 모놀리스 vs 마이크로서비스에서의 데이터 일관성

- 모놀리스 아키텍처
    - 여러 연산(잔여 객실 차감, 예약 생성 등)을 하나의 DB, 하나의 트랜잭션 안에서 처리할 수 있어 ACID 보장이 쉽다.
    - 그림 7.18처럼 “잔여 객실 서비스 + 예약 서비스”가 논리적으로 나눠져 있어도, 같은 DB를 쓰면 결국 한 트랜잭션으로 묶을 수 있다.

- 마이크로서비스 아키텍처(서비스별 독자 DB)
    - 각 서비스가 자기 DB를 가지면, 하나의 논리적 작업이 여러 DB 트랜잭션으로 쪼개짐.
    - 예: 잔여 객실 DB에는 성공, 예약 DB에서는 실패 → 데이터 불일치 발생 가능.
    - 즉, “한 번에 commit/rollback 되는 단일 트랜잭션” 개념을 쓰기 어려워지고, 데이터 일관성 문제가 생긴다.

### 분산 환경에서 일관성을 맞추는 대표 기법

2PC(2-phase commit)
- 여러 노드(여러 DB)에 걸친 분산 트랜잭션을 프로토콜로 보장하는 방식.
- 모두 성공/모두 실패를 보장하지만,
- 블로킹 특성 때문에 장애 시 복구까지 멈춰 있을 수 있고
- 성능·복잡도가 높아서 대규모 시스템에서는 잘 안 쓰인다.

사가(Saga) 패턴
- 큰 작업을 여러 개의 로컬 트랜잭션으로 쪼개고, 각 단계마다 “성공 시 다음 단계 트리거, 실패 시 보상(보정) 트랜잭션 실행”으로 맞추는 방식.
- 한 번에 ACID를 보장하는 대신, 일정 시간 안에 데이터가 수렴하는 eventual consistency(최종 일관성) 모델을 택한다.
- 로직·메시지 플로우가 복잡해져 설계 난이도와 시스템 복잡도가 크게 올라간다.

이 책(호텔 예약 시스템)에서의 결론

- 2PC나 사가 같은 분산 트랜잭션 메커니즘은 복잡도 대비 이득이 클 때만 선택할 가치가 있다.
- 이 예제의 호텔 예약 시스템에서는,
    - 예약/잔여 객실 정보를 같은 관계형 DB에 저장하는 하이브리드 방식이면 충분하다고 판단.
    - 즉, 마이크로서비스 “순수주의”보다는 실용성을 택해서 단일 DB의 트랜잭션(ACID)에 의존하는 설계를 선택했다.

### 마무리
이 장에서는 호텔 예약 시스템 전체 설계 과정을 한 번 훑었다.  
- 요구사항 수집 → 트래픽·데이터 규모 추정
- API 설계, 초기 데이터 모델, 시스템 아키텍처 다이어그램 작성

상세 설계 과정에서 “특정 방이 아니라 객실 유형을 예약한다”는 도메인 특성을 발견하고, 그에 맞게 DB 스키마와 설계를 수정했다.
동시 예약 상황에서 생기는 경쟁 조건을 다루기 위해  
- 비관적 락
- 낙관적 락
- DB 제약 조건(CHECK 등)

세 가지 접근을 비교·검토했다. 이후 DB 샤딩, Redis 캐시 등을 이용한 시스템 규모 확장 전략을 살펴보고,  
마이크로서비스 아키텍처에서의 데이터 일관성 문제와 대표적인 해결책들도 간단히 짚고 마무리했다.
