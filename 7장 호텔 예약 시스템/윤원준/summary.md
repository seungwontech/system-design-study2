**설계의 트레이드오프, 면접 질문, 심화 토론**을 위주로 진행하기 위한 정리 자료입니다.

## 1. 핵심 요약

### 1.1. 목표 및 요구사항 정의
*   **시스템 목표**: 5,000개 호텔, 100만 개 객실을 관리하는 대규모 호텔 체인의 예약 시스템 구축.
*   **핵심 기능**:
    *   사용자: 호텔/객실 정보 조회, 객실 예약 및 결제, 예약 취소.
    *   관리자: 호텔/객실 정보 관리, 초과 예약(Overbooking) 지원.
*   **주요 비기능 요구사항**:
    *   **높은 동시성(Concurrency)**: 성수기, 이벤트 기간 동안 특정 객실에 대한 동시 예약 요청을 안정적으로 처리해야 함.
    *   **데이터 일관성(Consistency)**: 이중 예약(Double Booking)이나 잘못된 결제가 발생해서는 안 됨.

### 1.2. 데이터 모델링: 왜 RDBMS인가?
*   **DB 선택**: NoSQL이 아닌 **RDBMS(MySQL/PostgreSQL)**를 선택.
    *   **이유**: 호텔 예약은 쓰기보다 읽기(조회)가 훨씬 많고, 무엇보다 **ACID 트랜잭션** 보장을 통해 데이터의 정합성을 지키는 것이 매우 중요하기 때문.
*   **핵심 스키마 설계**:
    *   **잘못된 설계**: `room` 테이블에 `status`를 두는 방식. (사용자는 특정 '객실'이 아닌 '객실 유형'을 예약함)
    *   **개선된 설계**: `room_type_inventory` 테이블을 도입하여 날짜별/객실 유형별 재고를 관리.
        *   **Primary Key**: `(hotel_id, room_type_id, date)`
        *   **핵심 컬럼**: `total_inventory` (총 재고), `total_reserved` (예약된 수)
        *   **동작 방식**: 날짜별, 객실 유형별로 재고를 미리 생성해두고, 예약이 발생할 때마다 해당 날짜의 `total_reserved`를 1씩 증가시킴.

    ![개선된 DB 스키마](https://drive.google.com/thumbnail?id=1sIA67gUffdkmk8EgAHXp10UsG8-6HsOY&sz=w700)

### 1.3. 아키텍처: 마이크로서비스 접근
*   시스템을 기능 단위로 분리하여 독립적인 개발 및 배포, 확장을 용이하게 함.
    *   **호텔/요금 서비스**: 호텔 정보, 객실 유형, 날짜별 요금 등 비교적 정적인 데이터를 제공 (캐싱에 유리).
    *   **예약 서비스**: 실제 예약 요청을 처리하고, 재고를 차감하는 핵심 로직 수행.
    *   **결제 서비스**: 외부 결제 게이트웨이와 연동하여 결제 처리.
    *   **호텔 관리 서비스**: 호텔 직원을 위한 내부용 관리 기능 제공.

    ![마이크로서비스 아키텍처](https://drive.google.com/thumbnail?id=1xSHM-K0E062C6nqd5rN1pjkOys5U53Oo&sz=w700)

### 1.4. 핵심 과제 1: 동시성 제어 (이중 예약 방지)
*   **문제 상황**: 남은 객실이 1개일 때, 두 명의 사용자가 동시에 예약 버튼을 누르는 경우 (Race Condition).
*   **해결 전략 (Trade-offs)**:
    1.  **비관적 락 (Pessimistic Lock)**: `SELECT ... FOR UPDATE` 구문으로 재고 레코드에 락을 검. 데이터 정합성은 완벽하지만, 락을 기다리는 동안 다른 트랜잭션이 대기하여 성능 저하 및 데드락 위험.
    2.  **낙관적 락 (Optimistic Lock)**: 테이블에 `version` 컬럼을 추가. 업데이트 시 `version`을 확인하여 충돌을 감지하고, 충돌 시 재시도. 락이 없어 성능은 좋지만, 경쟁이 심하면 재시도 비용이 급증.
    3.  **DB 제약 조건 (Constraint)**: `CHECK (total_reserved <= total_inventory)` 같은 제약 조건을 활용. DB 레벨에서 원천적으로 막아주지만, 유연성이 떨어지고 DB에 의존적.
*   **최종 선택**: 일반적인 상황에서는 **낙관적 락**이 성능과 구현 복잡도 사이의 합리적인 균형점.

    ![낙관적 락 동작 방식](https://drive.google.com/thumbnail?id=1jx4M3DkRm9x2PNO75GELZQ7uQM-QSr7e&sz=w600)

### 1.5. 핵심 과제 2: 확장성 확보
*   **DB 부하 분산**:
    *   **샤딩(Sharding)**: 대부분의 조회/예약이 `hotel_id`를 기준으로 일어나므로, `hotel_id`를 샤딩 키로 사용하여 DB를 수평 확장.
    *   **캐싱(Caching)**: 읽기 부하가 심한 **잔여 객실 수** 정보를 **Redis**에 캐싱.
        *   **주의**: 캐시는 조회용일 뿐, 실제 예약 시점에는 반드시 DB에서 최종 재고를 확인하여 일관성을 보장해야 함 (Source of Truth는 DB).

    ![캐시를 이용한 읽기 부하 분산](https://drive.google.com/thumbnail?id=1n7pzDz8rbeFFMvGRxDZrbPZNtnPI2m2I&sz=w600)

### 1.6. 심화 과제: MSA에서의 데이터 일관성
*   **문제 상황**: 예약 서비스(DB A)에서는 예약이 성공했는데, 결제 서비스(DB B)에서 실패하면 데이터가 불일치하게 됨.
*   **해결 방안**: 분산 트랜잭션 패턴 도입 고려.
    *   **2PC (2-Phase Commit)**: 모든 서비스가 동의해야만 최종 커밋. 강력한 일관성을 보장하지만, 특정 노드 장애 시 전체가 중단되어 가용성이 떨어짐.
    *   **Saga 패턴**: 각 서비스가 로컬 트랜잭션을 수행하고, 실패 시 **보상 트랜잭션(Compensating Transaction)**을 통해 이전 상태로 되돌림. 결과적 일관성(Eventual Consistency) 모델.

    ![MSA 데이터 일관성 문제](https://drive.google.com/thumbnail?id=1kg9K8LDdslokTnZIRqpUOX1gGjELW0gU&sz=w700)

## 2. 핵심 트레이드오프 및 설계 논의 (Trade-offs)

### 2.1. 동시성 제어: 비관적 락 vs 낙관적 락 vs DB 제약 조건
| 방식 | **비관적 락 (Pessimistic)** | **낙관적 락 (Optimistic)** | **DB 제약 조건 (Constraint)** |
| :--- | :--- | :--- | :--- |
| **동작** | 데이터를 읽을 때 락을 걸어 다른 접근 차단 | 락 없이 업데이트 시 버전을 확인하여 충돌 감지 | 업데이트 시 조건(`count >= 0`) 위반하면 롤백 |
| **장점** | 데이터 무결성 완벽 보장, 충돌이 매우 잦을 때 유리 | 락 오버헤드 없음, 충돌이 적을 때 성능 우수 | 구현이 매우 간단함 |
| **단점** | 성능 저하, 데드락 위험, 확장성 낮음 | 충돌 시 재시도(Retry) 로직 필요, 경쟁 심하면 성능 급락 | 유연성 부족, DB 의존적 |
| **적합성** | **금융, 재고 관리 (엄격한 정합성)** | **일반적인 예약 시스템 (적당한 경쟁)** | **간단한 재고 차감** |

> **논의 포인트**: 
> * "BTS 콘서트 날짜의 호텔 예약처럼 경쟁이 극심하다면?" (낙관적 락은 재시도 폭주로 DB를 죽일 수 있음. 비관적 락이나 Redis Lua Script 등을 고려해야 함)

### 2.2. 아키텍처: Monolith vs Microservices
| 구분 | **Monolith** | **Microservices (MSA)** |
| :--- | :--- | :--- |
| **트랜잭션** | **ACID 트랜잭션** 하나로 데이터 일관성 보장 쉬움 | 분산 트랜잭션(Saga, 2PC) 필요, **데이터 정합성 맞추기 어려움** |
| **확장성** | 특정 기능만 스케일아웃하기 어려움 | 서비스별 독립적 스케일아웃 가능 |
| **복잡도** | 낮음 | **매우 높음** (네트워크 지연, 장애 전파 등) |

> **논의 포인트**: 
> * "호텔 예약 시스템에서 MSA가 꼭 필요할까?" (예약과 결제는 강하게 결합되어야 하므로, 섣불리 찢으면 분산 트랜잭션 지옥에 빠질 수 있음. 예약+결제를 하나의 서비스로 묶는 것도 방법)

## 3. 면접 예상 질문 (Interview Q&A)

### Q1. "이중 예약(Double Booking)은 어떻게 방지하나요?"
* **핵심 키워드**: `트랜잭션 격리 수준`, `락(Lock)`, `유니크 제약 조건`, `멱등성 키`
* **답변 논리**:
    * **같은 사용자 중복 클릭**: 클라이언트 측 버튼 비활성화 + 서버 측 **멱등성 키(Idempotency Key)**를 사용하여 동일한 요청이 여러 번 처리되는 것을 방지합니다.
    * **다른 사용자 동시 예약**: RDBMS의 **트랜잭션**과 **낙관적 락(Optimistic Lock)** 또는 **비관적 락(Pessimistic Lock)**을 사용하여 재고 확인과 차감을 원자적(Atomic)으로 수행합니다.

### Q2. "Redis 캐시와 DB의 재고 데이터가 불일치하면 어떻게 하나요?"
* **핵심 키워드**: `최종적 일관성`, `DB가 진실의 원천(Source of Truth)`, `Check-and-Set`
* **답변 논리**:
    * 캐시는 조회 성능을 위한 것이지, 데이터의 원천이 아닙니다. **DB가 항상 진실의 원천(Source of Truth)**입니다.
    * 캐시에는 "재고가 있을 것 같다"는 정보만 주고, **실제 예약 시점에는 반드시 DB에서 재고를 다시 확인(Check-and-Set)**해야 합니다.
    * 재고가 없으면 사용자에게 "죄송합니다, 방금 매진되었습니다"라고 알리고, 비동기적으로 캐시를 갱신합니다.

### Q3. "100만 개의 객실 데이터를 어떻게 관리하나요?"
* **핵심 키워드**: `정규화`, `날짜별 재고 테이블`, `샤딩`
* **답변 논리**:
    * `room` 테이블과 `room_type_inventory` 테이블을 분리하여 관리합니다.
    * `room_type_inventory` 테이블에 `(hotel_id, room_type_id, date)`를 복합키로 하여 날짜별 재고를 미리 생성해둡니다.
    * 데이터가 많아지면 `hotel_id`를 기준으로 **샤딩**하여 부하를 분산합니다.

## 4. 심화 토론 주제 (Deep Dive Discussion)

### 주제 A: 분산 환경에서의 데이터 일관성 (Saga Pattern)
* **상황**: 예약 서비스는 성공했는데, 결제 서비스에서 잔액 부족으로 실패했다면?
* **접근법**:
    * **보상 트랜잭션(Compensating Transaction)**: 결제 실패 이벤트를 발행하면, 예약 서비스가 이를 수신하여 "예약 취소(롤백)" 로직을 실행해야 합니다. (Saga 패턴의 Choreography 또는 Orchestration 방식)

### 주제 B: 대규모 트래픽 대응 (Redis Lua Script)
* **상황**: 초당 수만 건의 예약 요청이 들어온다면 RDBMS만으로는 버티기 힘듭니다.
* **접근법**:
    * **Redis Lua Script**: 재고 확인과 차감을 Redis 내에서 원자적으로 실행하여 DB 부하를 줄입니다.
    * Redis에서 성공한 요청만 비동기(Kafka 등)로 DB에 반영하여 최종적 일관성을 맞춥니다. (단, Redis 데이터 유실 위험에 대한 대비 필요)

### 주제 C: 초과 예약(Overbooking) 전략
* **상황**: 호텔은 노쇼(No-Show)를 대비해 110% 예약을 받고 싶어 합니다.
* **접근법**:
    * `total_inventory`를 실제 객실 수보다 높게 설정하거나, 별도의 `overbooking_limit` 컬럼을 두어 로직에서 제어합니다.
    * 초과 예약 발생 시 대체 객실 제공이나 보상 정책 등 비즈니스 로직이 함께 고려되어야 합니다.
